# 生成式战法战术知识生成概念模型

---

## 一、研究背景

### 1.1 问题提出

当前战术战法知识的获取与应用面临三重困境：

1. **资料适配性不足**：现有战术战法资料聚焦于近代战争环境和人力作战条件，随着人工智能与无人作战平台的快速发展，传统战术知识难以直接迁移至现代无人协同作战场景。
2. **知识粒度断层**：现有战术文档多为"战役/战术级"宏观描述，缺乏面向机器人可执行的原子级行动指令，从"连排级战术"到"机器人可执行指令"之间存在显著的粒度鸿沟。
3. **场景覆盖有限**：已有的结构化战术知识库难以穷尽所有作战场景组合，面对未见过的新场景时缺乏有效的战术生成手段。

### 1.2 研究动机

大语言模型（Large Language Model, LLM）展现出强大的智能涌现能力，能够在给定上下文条件下生成结构化、逻辑连贯的专业内容。这为"从场景出发，自动生成战术知识"提供了技术可行性。然而，纯LLM生成存在两个核心风险：

- **知识幻觉**：LLM可能生成违反军事原则或物理规律的战术方案。
- **质量不可控**：生成结果的质量波动大，缺乏系统性的质量保证机制。

因此，需要构建一个**场景驱动、知识锚定、多智能体协同**的生成式战术知识生成概念模型，在充分利用LLM生成能力的同时，通过检索增强、规则约束和迭代精炼机制确保生成质量。

### 1.3 研究目标

本文提出一种**生成式战法战术知识生成概念模型**（Generative Tactical Knowledge Generation Conceptual Model, GTKG-CM），旨在：

1. 形式化定义作战场景空间与战术知识的表示方法，引入 USD（Universal Scene Description）作为标准化三维场景输入格式；
2. 建立从大规模真实 USD 场景文件到原子级战术知识的完整处理流水线，解决"928MB Blender 导出场景 → 子场景提取 → 战术生成"的工程挑战；
3. 建立从 USD 场景到战术知识的自适应生成策略（检索增强生成与自主生成的动态切换）；
4. 设计多智能体协作的迭代精炼框架，保证生成战术的质量与军事可行性；
5. 构建结构化的战术质量评价体系，支持战术知识的分级管理与持续优化。

> **当前实现技术栈**：Python 3.10+，LLM 推理使用 MiniMax API（MiniMax-M2.7 模型），嵌入模型支持 OpenAI `text-embedding-3-large`（在线）和 `bge-large-zh-v1.5`（本地），向量数据库使用 ChromaDB PersistentClient。完整代码位于 `src/` 目录下，各阶段独立为子包，通过 `src/pipeline.py` 中 `PipelineRunner` 协调器串联。代码库当前包含 5 个单元测试文件。

## 二、相关概念定义

### 2.0 基本设计原则

在定义具体概念之前，明确以下设计原则：

1. **编组化描述原则**：战术中的执行主体使用功能性角色（如"压制组"、"侦察组"、"侧翼警戒组"）描述，而非具体机器人编号（如"人形机器人1号"、"无人机2号"）。战术描述的是角色之间的**协同模式**，而非兵力组成——"需要多少人"是部署决策，不是战术设计决策。此原则确保战术的通用性，使其不绑定于特定机器人个体，可在不同编队规模下灵活运用。

2. **双版本输出原则**：每条战术生成两个版本——**文字描述版**（侧重可读性与VLA理解）和**结构化描述版**（侧重可执行性与机器人行动序列）。文字描述版对应粗粒度，结构化描述版对应细粒度。

3. **参数占位原则**：结构化描述版中的精确参数（坐标、速度、射速等）在生成阶段留为占位符，由执行阶段根据实际环境填入，战术生成阶段仅定义动作逻辑和执行框架。

4. **场景无关性原则（三重不变性）**：战术描述的是行动模式，必须具有**三重不变性**——(1) **旋转不变性**：将场景绕任意轴旋转任意角度后，战术文本中的每一个字依然完全正确。绝对方向绑定场景的偶然朝向，关系描述绑定空间的不变拓扑——前者随场景旋转而改变，后者在任何朝向下都成立。(2) **物体功能不变性**：将场景中的掩体/障碍物替换为功能等价但外观不同的物体（如"低矮木质长桌"→"低矮方形花坛"）后，战术文本中的每一个字依然完全正确。具体物体身份绑定场景的偶然陈设，功能类别描述绑定物体的不变战术作用。(3) **编队规模不变性**：将编队中的单元数量任意增减（2人→3人→4人）后，战术文本中的编组交互描述依然完全正确。战术定义的是编组交互模式（"前方掩护组与后方掩护组的交替掩护关系"），而非兵力组成（"两个机器人一前一后"）。战术中不得出现具体数值（距离、数量、时间），不得出现场景特定物体身份描述（家具/设施类别如桌子、沙发、椅子、花坛、空调外机，以及材质品牌如木质、樱桃木、不锈钢——必须使用战术功能类别如"低矮掩体"、"大型遮挡物"、"柱状掩体"），不得出现绝对方向指代（如"走廊西端"、"北侧开口"、"北墙"——必须使用功能/关系描述如"走廊远端"、"走廊一侧的开口"、"走廊一侧墙壁"），不得出现编队规模限定（如"双人编队"、"两名突击组"、"三人战斗小组"、"1名掩护组+1名突击组"——必须使用编组交互描述如"编队"、"突击组与掩护组协同"）。

5. **相对关系描述原则**：战术中的空间关系使用相对空间关系词（紧邻/近距离/中距离/远距离），时间关系使用相对时间词（即时/短暂延迟后/同步/交替），火力强度使用定性描述（单发精准射击/短点射/长点射/持续压制），而非具体数值。具体参数在执行阶段由场景绑定机制注入。

6. **USD 统一场景输入原则**：作战场景的全部信息——三维空间布局、建筑结构、掩体分布、实体标记（敌人/人质）、环境条件——统一通过单个 USDA 文本文件描述，不再维护独立的语义元数据层。一切场景属性（敌人威胁程度、人质状态、光照与天气）均以 USD Prim 和自定义属性的形式嵌入 USDA 文件，而非通过外部 JSON 或结构化维度提供。唯一的额外输入是**任务描述文本**（1-3句自然语言），用于传达指挥官意图和任务约束——这些是指挥决策，不是场景属性，无法从几何场景中推断。

> **当前工程现实与理想目标的差距**：上述原则描述的是 **GTKG-CM 的目标架构**。当前从 Blender 导出的 USDA 文件不含 `entity_type`、`threat_level` 等语义标注（详见附录 B.6），因此 Phase 1 的全局场景理解无法直接从 USD 读取实体信息。作为过渡方案，Phase 3c 通过 LLM 从几何和空间关系中推理潜在威胁（`inferred_threats`），Phase 4 M1 将这些推理结果作为实体信息使用。未来引入支持 customData 语义标注的 USD 创作管线后，实体信息将直接从 USDA 读取，无需 LLM 推理。

#### 2.0.1 相对关系词汇库

**空间关系词汇**：

| 词汇 | 含义 | 执行阶段绑定范围 |
|------|------|-----------------|
| 紧邻 | 贴靠掩体，1米以内 | 0.5-1m |
| 近距离 | 可快速抵达，无需跃进 | 1-3m |
| 中距离 | 需要跃进或掩护机动 | 3-10m |
| 远距离 | 需要多段掩护机动 | 10-30m |

**时间关系词汇**：

| 词汇 | 含义 | 执行阶段绑定范围 |
|------|------|-----------------|
| 即时 | 无延迟，立即执行 | 0s |
| 短暂延迟 | 制造战术时间差 | 0.3-1s |
| 同步 | 同一时刻执行（需同步信号） | 0s（同步信号触发） |
| 交替 | 一个动作完成后触发下一个 | 动作完成事件触发 |

> **注意**："同步"和"交替"本质上是事件触发型时间关系——"同步"依赖同步信号的到达时刻，"交替"依赖前一动作的完成事件。在场景绑定机制中，这两类词汇绑定的是**触发条件**而非具体时间数值，与"即时"和"短暂延迟"（绑定数值化时间延迟）的处理方式不同。

**火力强度词汇**：

| 词汇 | 含义 | 执行阶段绑定范围 |
|------|------|-----------------|
| 单发精准射击 | 精确瞄准后的单发射击 | 1发 |
| 短点射 | 短暂连发射击 | 2-3发 |
| 长点射 | 较长连发射击 | 5-8发 |
| 持续压制 | 连续火力覆盖 | 10发以上/连续射击 |

#### 2.0.2 方向泛化词汇库

战术描述中禁止使用绝对地理方向词（东/南/西/北），必须替换为以任务轴线为基准的功能/关系描述。

**方向替换对照表**：

| 绝对方向（禁止） | 功能/关系替换（使用） |
|---|---|
| 走廊西端 / 走廊西段 | 走廊远端 / 走廊目标方向端 / 走廊尽头 |
| 走廊东端 / 走廊东段 | 走廊近端 / 走廊入口方向段 |
| 北侧开口 | 走廊中部侧向开口 / 走廊一侧的开口 |
| 北侧墙壁 | 走廊目标侧墙壁 / 走廊一侧墙壁 |
| 南侧墙壁 | 走廊入口侧墙壁 / 走廊对侧墙壁 |
| 东侧入口 | 走廊近端入口 / 机动起点方向入口 |
| 西侧入口 | 走廊远端入口 / 目标方向入口 |
| 朝南 / 朝北 / 朝东 / 朝西 | 朝向入口方向 / 朝向目标方向 / 朝向走廊一侧 |
| 北墙 / 南墙 | 走廊一侧墙壁 / 走廊对侧墙壁 |
| 东墙 / 西墙 | 走廊近端墙壁 / 走廊远端墙壁 |
| 北墙尽头 / 北墙墙角 | 走廊一侧墙壁尽头 / 走廊一侧墙壁转角 |
| 东角 / 西角 / 北角 / 南角 | 近端转角 / 远端转角 / 侧向转角 / 对侧转角 |

**通用方向替换原则**：

1. **沿任务轴线方向** → 近端/远端（以编队出发点为"近"，目标点为"远"）
2. **垂直于任务轴线** → 一侧/对侧（以编队面向为参照）
3. **垂直方向** → 上层/下层（不引用具体楼层号）
4. **入口/开口位置** → 以功能描述（"通往楼梯的入口"、"中段侧向开口"）
5. **空间段落** → 以功能分段（"入口段"、"中段"、"目标段"）

> **设计动机**：desc.json 作为场景语义标注，天然包含绝对方向信息（如 `Z1-COR-EAST`、`Z1-COR-WEST`）、场景特定物体描述（如"低矮长桌"、"大型L形沙发"）和编队规模隐含假设。若不做预处理直接将 desc.json 送入 LLM，生成战术会自然"借用"这些方向、物体身份和编队规模词（源头污染），导致战术绑定到特定场景的地理方向、物体陈设和兵力组成。为此 Phase 4 引入**三层防线体系**：(1) `direction_generalizer.py` 预处理模块——在 desc.json 送入 LLM 前自动将绝对方向替换为功能/关系描述（`generalize_directions()`）+ 将物体身份替换为战术功能类别（`generalize_objects()`）+ 将编队规模限定替换为编队/角色通用描述（`generalize_team_size()`），从源头切断方向、物体身份和编队规模三重污染；(2) A_gen 的 system prompt 中注入**三重不变性哲学**和方向/物体功能/编队规模泛化词汇库（`DIRECTION_VOCABULARY` + `OBJECT_FUNCTIONAL_VOCABULARY` + `TEAM_SIZE_VOCABULARY`），教会 LLM 用功能/关系语言、战术功能类别和编组交互描述思考，而非机械避开违禁词；(3) A_review 的 G-T2 硬约束（含编队规模数字词"双人"/"双组"等）+ G-T7 硬约束（物体身份禁止）+ G-T10 硬约束（方向禁止）+ G-T11 场景标识符禁止作为最终语义审查安全网。

### 2.1 战术知识的形式化定义

基于已有的战术模板规范，战术知识在逻辑层定义为一个多元组：

$$T = \langle \text{ID}, \text{Meta}, \text{Desc}, \text{Seq}, \text{Vis}, \text{Q} \rangle$$

该多元组是战术知识的**逻辑抽象**，其具体实现采用JSON格式（详见战术模板v1.0）。每条战术输出**两个版本**：

- **文字描述版**（粗粒度）：$\text{Seq}$ 中的行动以自然语言详细描述，配合视觉辅助说明，侧重可读性与VLA模型理解
- **结构化描述版**（细粒度）：$\text{Seq}$ 中的行动以结构化指令序列描述，参数留为占位符，侧重可执行性与机器人直接理解

两个版本共享相同的 $\text{ID}$、$\text{Meta}$、$\text{Desc}$、$\text{Vis}$、$\text{Q}$，仅 $\text{Seq}$ 的格式不同。

多元组各分量与JSON字段的映射关系如下：

| 符号 | 对应字段 | 类型 | 说明 |
|------|----------|------|------|
| $\text{ID}$ | `Tactic_ID` | String | 战术唯一标识符，格式如 `CQB_04` |
| $\text{Meta}$ | 元数据集合 | Object | 包含 `Tactic_Name`、`Mission_Phase`、`Tactic_Type`、`Applicable_Environment`、`Execution_Time`、`Parent_Tactic`、`Sub_Tactics`、`Semantic_Tags`、`Source_Sub_Scene` |
| $\text{Desc}$ | `Objective` + `Description` | String | 战术目标与详细描述，执行主体使用编组化描述（如"压制组"、"侦察组"），不绑定具体机器人编号 |
| $\text{Seq}$ | `Action_Sequence` | Array | 有序行动序列 $[a_1, a_2, \ldots, a_n]$。文字描述版：$a_i = \langle s_i, \text{intent}_i^{\text{rich}}, \text{vis}_i \rangle$；结构化描述版：$a_i = \langle s_i, \text{intent}_i^{\text{brief}}, \text{instr}_i \rangle$ |
| $\text{Vis}$ | `Visual_Aid_Overall` | Array | 整体战术示意图（路径或文字描述） |
| $\text{Q}$ | `Credibility` | Float | 质量评分 $[0, 10]$，由评价体系生成 |

**行动原子（Action Atom）** 根据版本不同有两种形式化定义：

**文字描述版行动原子**：

$$a_i^{\text{text}} = \langle s_i, \text{intent}_i^{\text{rich}}, \text{vis}_i \rangle$$

其中 $s_i$ 为步骤序号，$\text{intent}_i^{\text{rich}}$ 为自然语言的详细战术意图描述（100-300字），$\text{vis}_i$ 为对应的视觉辅助描述。执行主体使用编组化描述（如"压制组"、"侦察组"），不绑定具体机器人编号。

**结构化描述版行动原子**：

$$a_i^{\text{struct}} = \langle s_i, \text{intent}_i^{\text{brief}}, \text{instr}_i \rangle$$

其中 $\text{intent}_i^{\text{brief}}$ 为简要意图标题，$\text{instr}_i$ 为结构化指令序列，包含角色标识、动作类型、参数占位符等。精确参数（坐标、速度、射速等）在生成阶段留为占位符，由执行阶段根据实际环境填入。

### 2.2 作战场景的形式化定义

作战场景在 GTKG-CM 中通过 **USD（Universal Scene Description）文件**统一描述。场景的全部信息——三维空间布局、建筑结构、掩体分布、实体标记、环境条件——均以 USD Prim 和自定义属性的形式嵌入单个 USDA 文本文件。

> **Prim（场景图基本节点）**：Prim 是 USD 场景图中的基本组成单元，全称 Primitive。每个 Prim 代表场景中的一个可寻址对象——可以是一个几何体（Mesh）、一个变换节点（Xform）、一个光源（Light）、一个材质（Material），或一个逻辑分组容器（Scope）。Prim 按层级嵌套形成场景图（Scene Graph），每个 Prim 可携带属性（Property）和自定义元数据（customData）。在战术场景中，"一堵墙""一个敌人标记""一个掩体"分别对应一个 Prim。详见附录 B.2。

#### 2.2.1 两类场景：大场景与子场景

实际工程中，场景分为两个层级：

| 层级 | 来源 | 规模 | 用途 |
|------|------|------|------|
| **大场景**（Global Scene） | Blender/Maya 等 DCC 工具导出 | 数百MB–GB，数万Prim | 完整建筑+外场建模，无法直接输入LLM |
| **子场景**（Sub-Scene） | 从大场景几何裁切+语义简化 | 5-50KB，≤60个Cube Prim | LLM可直接消费的战术级场景片段 |

大场景需经过预处理流水线（Phase 0-3）转化为子场景后，方可进入 GTKG-CM 核心生成流程（Phase 4，即 M1-M4）。

> **楼层编号约定**：全文采用以下统一规则——`F_ground` = 外场地形（Z 最低层），`F1` = 建筑地面层（一楼，Z≈0.5–3.8m），`F2` = 建筑二层（Z≈3.8–6.8m），`F3` = 建筑三层（Z≈6.8–10.0m）。中文语境下"一层"对应 F1（地面层），"二层"对应 F2，以此类推。子场景的 `floor` 字段遵循此编号。

#### 2.2.2 场景的 USD 表示

一个完整的战术场景 USDA 文件包含地形、建筑结构（墙体/楼板/屋顶）、开口（门/窗/阳台）、掩体、光源、敌方/人质标记和环境效果等要素。各 USD 要素到战术空间的完整映射关系（含 Prim 类型、提取方式和当前可用性）详见附录 B.5。

#### 2.2.3 任务描述

除 USD 文件外，**唯一的额外输入**是**任务描述文本** $\text{task}$（1-3句自然语言），用于传达 USD 场景无法表达的**指挥官意图**和**任务约束**：

```
任务描述示例：
"进攻该三层建筑，解救人质。已知二楼和三楼有敌方守卫，人质位置不确定。
约束：优先保证人质安全，尽量俘虏敌人而非击毙。"
```

任务描述包含两类信息：
- **作战意图**（侦察/进攻/防御/撤退）：这是指挥官的决策，不是场景的物理属性
- **任务约束**（人质优先、俘虏优先、时间窗口等）：这是行动规则，无法从几何场景中推断

#### 2.2.4 场景的形式化定义

$$s = \langle \text{USD}, \text{task} \rangle$$

其中：
- $\text{USD}$ 为 USDA 格式的三维场景描述文本，包含全部空间信息、实体标记和环境条件
- $\text{task}$ 为自然语言任务描述（1-3句话），指明指挥官意图和任务约束

LLM 读取 USDA 文件后，自动生成**全局场景描述文本** $\text{desc}_{\text{global}}(s)$（200-400字的自然语言场景摘要，由 Phase 1 产出），用于后续检索和生成模块的文本输入。同时，LLM 自动推断场景标签（如"建筑三层"、"开阔地接近"、"夜间"等），用于知识库检索时的场景匹配。标签为自由形式，不强制限定在预定义的维度取值集合内。

> **与子场景描述的区别**：此处 $\text{desc}_{\text{global}}(s)$ 是 Phase 1 产出的大场景级全局描述（200-400字）。Phase 3c 产出的 `spatial_description` 是子场景级描述（100-200字，见 §7.3）。两者粗细粒度不同，不应混淆。

> **注意**：此处描述的是 GTKG-CM 的概念模型（理想流程）——LLM 直接从 USDA 理解场景。在实际工程流水线中，场景理解是分层递进的：Phase 1 生成全局场景理解（大场景级，侧重建筑整体结构、楼层连接关系、主要出入口），Phase 3c 通过 LLM 生成每个子场景的 desc.json（子场景级，包含 zones/openings/cover_assessment/threats/movement 完整语义标注），Phase 4 M1 对 desc.json 进行一致性验证并补充战术标注（火力扇区、同步点等，详见 §8.2）。三层递进并非冗余：Phase 1 侧重"建筑整体是什么样的"，Phase 3c 侧重"这个局部区域如何被战术利用"，M1 侧重"这些语义标注是否自洽、还需要补充什么战术信息"。

### 2.3 生成式战术知识生成的核心问题

给定：
- 作战场景 $s \in \mathcal{S}$（USD 场景文件 + 任务描述文本）
- 已有战术知识库 $\mathcal{K} = \{T_1, T_2, \ldots, T_N\}$
- PDF参考资料集合 $\mathcal{R}$

求解一个生成函数：

$$G^*: \mathcal{S} \times \mathcal{K} \times \mathcal{R} \rightarrow \mathcal{T}$$

其中：
- $\mathcal{S}$ 为场景空间（USD + 任务描述）
- $\mathcal{K}$ 为已有战术知识库，用于检索相似场景的已验证战术作为参考（知识复用）
- $\mathcal{R}$ 为 PDF 参考资料集合，提供军事原则和战术教范知识（权威知识锚定）

使得输出战术 $T = G^*(s, \mathcal{K}, \mathcal{R})$ 满足：
1. **场景适配性**：$T$ 与场景 $s$ 高度匹配；
2. **军事可行性**：$T$ 不违反军事原则与物理约束；
3. **可执行性**：$T$ 的行动序列可被机器人直接理解和执行。

> **注意**：在当前实现中，M2 自适应生成策略的检索对象为 $\mathcal{R}$（PDF 参考资料），因为当前知识库 $\mathcal{K}$ 尚在构建中，尚未积累足够的已验证战术。随着知识库规模增长，M2 的检索范围可扩展为 $\mathcal{K} \cup \mathcal{R}$（知识库 + PDF 双源检索），即 8.3 节所述的知识检索策略的自然扩展。

## 四、Phase 0：场景预处理（Python 流式解析器）

### 4.0 预备知识：USD/USDA 与 3D 场景数据

在深入 Phase 0 的算法细节之前，需要先理解几个核心概念。

#### 4.0.1 USD/USDA 文件是什么

**USD**（Universal Scene Description）是 Pixar 发明的一种 3D 场景描述格式，目前已成为工业标准（NVIDIA Omniverse、Apple Vision Pro、Unreal Engine 均原生支持）。**USDA** 是其 ASCII 文本版本（A = ASCII），人类可以直接用文本编辑器打开阅读，不需要任何专用工具。

可以把它想象成一个巨大的"**3D 世界说明书**"，用嵌套的树形结构描述场景中的一切：

```
def Xform "building"                       # 一个叫 "building" 的容器
{
    def Cube "wall_north"                  # 里面有一堵墙，它是一个长方体
    {
        float3[] extent = [(-500,0,-20), (500,0,220)]   # 长宽高范围
        matrix4d xformOp:transform = (1,0,0,0, ...)     # 位置/旋转/缩放
        rel material:binding = </materials/concrete>     # 材质=混凝土
    }

    def Cube "door_01"                     # 旁边有一扇门
    {
        float3[] extent = [(-80,0,-5), (80,210,5)]
        matrix4d xformOp:transform = (...)
    }
}
```

整个场景就是一个树形结构：`世界 → 建筑 → 楼层 → 房间 → 墙体/门/窗/家具`，每一层都是一个节点（Prim）。

USD 将场景的全部信息——三维空间布局、建筑结构、掩体分布、材质、光照——统一嵌入单个文件，无需外部配置文件。

#### 4.0.2 Prim 是什么

**Prim**（"primitive" 的缩写）是 USD 场景树中的**一个节点**。类比文件系统：

| 文件系统 | USD 场景 |
|---------|----------|
| `/` 根目录 | `/` 根 Prim |
| `/home/` 子目录 | `/building` Xform Prim（容器，不含几何体，只做位置变换） |
| `/home/document.txt` 文件 | `/building/wall_north` Cube Prim（实际几何体） |

USDA 中有多种类型的 Prim：

| Prim 类型 | 包含什么 | Phase 0 的处理方式 |
|-----------|---------|-------------------|
| **Cube / Mesh** | 实际几何体（墙体、楼板、门、窗、家具...） | **提取**几何数据——战术分析需要知道空间中有哪些物体 |
| **Xform** | 空容器，定义子节点的位置/旋转/缩放 | **只用于累乘变换矩阵**（确定子节点的世界位置），本身不输出 |
| **Material** | 材质定义（颜色、纹理引用） | **提取颜色**帮助理解物体材质（混凝土/木材/玻璃），**跳过纹理贴图文件**（.png/.jpg 只是渲染用的表面装饰，不反映物理材质） |
| **Light / Camera** | 灯光、摄像机 | **跳过**——对战术分析无意义 |
| **Scope** | 组织性容器 | **跳过**——只是用来分组的逻辑节点 |

从每个几何 Prim 中提取的关键属性：

| 属性 | 含义 | 举例 |
|------|------|------|
| `prim_type` | 类型 | `"Cube"`（长方体）、`"Mesh"`（三角网格） |
| `extent` | 局部包围盒 | `[(-5, 0, -0.2), (5, 3, 0.2)]` — 在自身坐标系下的长宽高 |
| `transform` | 4×4 变换矩阵 | 包含平移、旋转、缩放，决定物体在场景中的真实位置 |
| `material:binding` | 材质引用路径 | `"/materials/concrete"` — 混凝土 |

#### 4.0.3 为什么跳过纹理贴图

USDA 的材质定义块中可能引用纹理图片文件：

```
def Shader "shader"
{
    asset inputs:diffuseTexture = @./textures/concrete_wall.png@
    asset inputs:normalMap      = @./textures/concrete_wall_norm.png@
}
```

这些 `.png` 贴图**不对战术分析提供有效信息**：

1. **纹理是表象，不是本质**：贴了砖纹贴图的石膏板墙，纹理文件名叫 `brick.png`，但物理上它仍然是石膏板，弹道防护能力接近于零。看纹理文件名判断材质会误判。
2. **材质名已经给出语义标签**：`material:binding = </materials/concrete>` 直接告诉我们设计者意图——这是混凝土，比纹理图推断更可靠。
3. **二进制大文件**：加载数千张纹理图只会增加处理负担，而 USD 的 `material:binding` 已经用一行文本表达了相同级别的信息。
4. **LLM 不看图片**：当前流水线的 LLM 只接收文本输入，纹理图无法直接送入。

> **保留的颜色信息**：`color3f diffuseColor = (0.8, 0.2, 0.2)`（红色）已被 Phase 0 提取。颜色对 LLM 战术分析有辅助价值——红色物体可能是警示标记或灭火器，灰白色是典型混凝土墙体，棕色是木质结构。

#### 4.0.4 局部坐标 vs 世界坐标 — 为什么需要世界包围盒

这是 Phase 0 最关键的几何操作。USD 文件中的每个几何体定义的 `extent` 是**局部坐标**下的包围盒，但后续处理需要知道物体在**整个场景中的真实位置**。

**类比**：在自己房间里测量一张桌子 —
- **局部坐标**：桌子长 2m × 宽 1m × 高 0.8m。这是桌子"自己"的尺寸。
- **世界坐标**：桌子在房间东北角，离北墙 0.5m，离东墙 1m。

USD 使用**父子层级变换**表达位置关系：

```
房间 A（transform = 平移(10, 0, 5)）
  └── 桌子（transform = 平移(1, 0, 0.5)）
       → 桌子的世界位置 = (10+1, 0+0, 5+0.5) = (11, 0, 5.5)
```

**世界包围盒（World BBox）的计算过程**：

1. 取局部 extent 的 **8 个角点**：`(x_min,y_min,z_min)`, `(x_min,y_min,z_max)`, ... 共 8 个
2. 用从根节点到该 Prim 的**累积变换矩阵**依次变换每个角点 → 8 个世界空间角点
3. 对 8 个角点的 x/y/z 分别取 min/max → 世界空间轴对齐包围盒（AABB）

```
局部 extent  = [(-1, -0.5, 0), (1, 0.5, 0.8)]
经父节点平移(10,0,5) + 旋转 45° 后：
  8个角点在世界空间的新位置 =
    (-0.35,-0.35,5.0), (1.06,-1.06,5.0), ...
世界包围盒 = {x:[-0.35,1.06], y:[-1.06,1.06], z:[5.0,5.8]}
```

这个操作对于后续所有阶段至关重要 —— Phase 3 需要世界坐标来判断"这个物体在 SS_01 的边界内吗？"，Phase 3c LLM 需要世界坐标来理解"走廊北端有一个转角"这样的空间关系。

在数学上，这是一个**4×4 齐次变换矩阵沿场景树自顶向下累乘**的过程。每个 Xform Prim 的 `xformOp:translate`、`xformOp:rotateXYZ`、`xformOp:scale` 被组合为一个 4×4 行优先矩阵 $M_i$，场景树中深度为 $d$ 的 Prim 的世界变换矩阵为：

$$T_{\text{world}} = M_0 \cdot M_1 \cdot M_2 \cdot \ldots \cdot M_d$$

其中 $M_i$ 由子节点的本地平移/旋转/缩放组合而成，乘法的顺序保证了子节点在父节点空间中的位置被正确累加。

#### 4.0.5 几何模式检测

Phase 0 不判断每个物体"是什么"（墙还是门），但会检测物体在**空间中排列的规律**。共检测三种模式：

**1. 等距线性排列**

```
□ □ □ □ □          ← 一排等间距的柱子/窗户
```

沿 x/y/z 轴排序所有 Cube 的中心坐标，滑窗检查相邻间距是否一致（偏差 < 10%）。若发现 ≥ 3 个物体等距排列，向外扩展直到间距跳变。置信度 = min(1.0, 元素数/10)。

→ *Phase 1 LLM 看到"走廊两侧各有一排等距 2m 的物体"，可以推断这是"柱廊"，对应特定战术模式。*

**2. 对称对**

```
  □     □           ← 门两侧对称的窗户
```

计算所有物体在对称轴上的坐标均值作为参考中心，只在轴一侧搜索。对每个物体计算其镜像位置，在同一空间哈希网格单元内查找是否存在于镜像位置的另一物体。非对称轴坐标差 ≤ 0.5m 则配对。

→ *对称是建筑设计的普遍规律，帮助 LLM 理解空间结构（"门两侧各有一扇窗"）。*

**3. 密集簇**

```
  □□□
  □□□□              ← 堆积在一起的杂物/家具/碎片
   □□
```

3D 空间哈希网格（cell_size = 0.5m）+ BFS 连通分量。距离 ≤ 0.5m 的物体视为相邻，连通分量 ≥ 4 个物体即为密集簇。

→ *密集簇可能代表"家具堆"、"废墟"、"障碍区"，影响机动路径规划。*

所有三种模式检测均使用**空间哈希优化**，将 O(n²) 暴力比较优化为 O(n·k)（k = 每个邻域内平均物体数），支持 31,620 个 Prim 全量检测不采样。

Blender 等 DCC 工具导出的 USDA 文件（928MB，95万行，31,620个Mesh）具有以下特征，使其无法直接供 LLM 消费：

1. **体积过大**：远超 LLM 上下文窗口（即使 1M token 模型也无法容纳）
2. **语义缺失**：对象名全为 Blender 自动生成（`Group_24__31772_278`、`__5847_.028`），无法语义识别
3. **几何冗余**：99% 体积为顶点坐标、法线、UV、面索引数据，对战术理解无直接价值
4. **坐标嵌套**：所有 Xform 共享相同的 rotateXYZ=90°（X轴旋转），需正确计算世界空间坐标
5. **引用困境**：`pxr.Usd` 库无法在 32GB 内存内完整加载该文件

### 4.2 解析策略

自建流式文本解析器，逐行读取 USDA，不依赖 pxr.Usd 库：

**步骤 1：逐行扫描 + brace counting 跟踪层级**

遇到 `def Xform/Mesh/Material/Scope` 时入栈，记录：
- Prim 名称、类型、层级路径
- `float3[] extent`（本地包围盒）
- `xformOp:translate / rotateXYZ / scale`（变换操作）
- `rel material:binding`（材质引用）

**步骤 2：几何数据整块跳过**

遇到以下属性时，跳过直到下一个非缩进属性（占 99% 体积）：
- `point3f[] points`
- `normal3f[] normals`
- `texCoord2f[] primvars:st`
- `int[] faceVertexCounts`
- `int[] faceVertexIndices`

**步骤 3：世界空间包围盒计算**

每个 Prim 出栈时，根据其本地 `extent` 和从根到该 Prim 的累积变换矩阵，计算世界空间包围盒。

变换累积的一般形式（每个 Xform 独立记录 translate 和 rotateXYZ）：

$$T_{\text{world}} = M_0 \cdot M_1 \cdot M_2 \cdot \ldots \cdot M_n$$

其中 $M_i = T_i \cdot R_i \cdot S_i$（translate × rotateXYZ × scale），$M_0$ 为根变换。

**本 USDA 文件的特殊性**：该 Blender 导出文件的所有 Xform 恰好共享相同的 `rotateXYZ = (90, 0, 0)`（绕 X 轴旋转 90°），即 $R_i = R_X(90°)$ 对所有 $i$ 成立。简化后的累积公式为 $T_{\text{world}} = T_{\text{root}} \cdot R_X(90°) \cdot T_1 \cdot R_X(90°) \cdot T_2 \cdot \ldots$。对于其他来源的 USDA 文件（如 Maya、Houdini、NVIDIA Isaac Sim），需从每个 Xform 的 `xformOp:rotateXYZ` 属性读取独立的旋转参数，不可硬编码此简化公式。

本地包围盒的 8 个角点经完整变换矩阵变换后，取 min/max 得世界空间包围盒。

**步骤 4：楼板面检测与楼层推断**

`classify_fragments` 将碎片按尺寸比分为 `vertical_walls` / `horizontal_planes` / `other`。从 `horizontal_planes` 中经 `extract_floor_planes`（tolerance=1.0m，相邻 Z 组间距≤1m 合并为同一楼板面，≥5碎片组成候选）提取楼板面候选。经 `_merge_adjacent_candidates`（threshold=6.0m，间距<6m 视为同一逻辑楼板的子结构——夹层/不同房间标高/台阶平台）二级合并得到逻辑楼板层级（slabs）。最后 `_build_stories_from_slabs` 按 **N−1 逻辑**生成可居住楼层（stories）：N 块楼板 = N−1 个楼层空间（最低板=地面层地板，中间板=下层天花板+上层地板，最高板=屋顶）。

**典型流程**：8 个楼板面候选（Z=182~247，evidence=1 strong + 7 medium）→ 间距<6m 合并为 4 个楼板层级 → N−1=3 个可居住楼层（F0/F1/F2）。

**步骤 5：墙体线与建筑特征检测**

`extract_wall_lines` 从 `vertical_walls` 中提取墙面线（同级 X/Y 坐标分组，间隙>0.5m=可能的门/窗开口，本场景产出 66 条 X 向 + 46 条 Y 向）。在此基础上运行 `detect_rooms_by_story`（先按楼层 Z 范围筛选墙面线，再运行正交墙面交点网格房间检测，≥3 面围合）、`detect_corridor_candidates`（平行墙面间隙中的长条形低密度矩形，宽度 1–5m、长度≥3m、密度<1.0 cubes/m，置信度由长宽比/密度/楼层匹配/房间连接决定）、`detect_stairwells_by_density`（相邻楼层间紧凑 XY 聚类（1–15 m²）+ 垂直密度≥0.3 的楼梯井候选）。

**步骤 6：空间预聚类**

按水平面连通分量分析，基于密度和 cube 比例将场景分区：`building`（占比≥15%+Z密度≥50）、`exterior_structure`（占比≥10%）、`outdoor_terrain`（低密度大范围）、`background`（<5%噪声）。

### 4.3 设计决策：不做启发式语义分类

**早期版本**试图通过 9 条几何规则从 triangulated mesh 碎片中推断每个 Prim 的语义类型（墙体/门/窗/掩体/柱/装饰等）。工程实践表明这在碎片化的三角网格上**不可靠**：

- 一面完整的墙被 Blender 三角化后产生数百个细碎 Mesh（如 0.2m × 0.02m × 8.5m 的细条），单独看每个碎片都不满足"墙"的几何判断（宽度 0.2m < 0.5m 阈值 → 不被识别为墙）
- 一整扇门由 50 个三角面组成，每个面高仅 3-5cm → "门高 1.8-2.5m"的规则一个碎片也匹配不上 → 0 个门被检测到
- 楼板被切成数十片，每片面积远小于 4m² → 只有 3 片被识别为楼板（实际应有 10+ 层楼板）
- 细长墙碎片（门框边缘、窗框竖条）满足"高+截面小"被误标为柱 → 13562 个误检 vs 实际柱子可能不到 100 个

**根本原因**：启发式规则假设每个 Prim 对应一个完整建筑构件（一面墙、一扇门），但 Blender 导出的 triangulated mesh 将每个构件切成了数十至数百个碎片。

**当前设计**：Phase 0 **仅输出纯几何数据**，不做语义分类标注。语义理解全部交给 Phase 1 LLM（$A_{\text{parse}}$），由 LLM 从 31620 个几何碎片的**空间分布**中推理建筑结构。LLM 擅长发现"Z=0 附近有一大片共面的细碎片 = 地面/楼板"、"X=5 处有一条 Z 向延伸的密集碎片带 = 墙"这类空间模式——这恰恰是规则系统不擅长的。

> **代码实现说明**：`src/phase0/heuristic_classifier.py` 文件保留在代码库中（包含 9 条几何规则和 TacticalType 枚举），但当前 `pipeline.py` 的 `run_phase0()` 不调用该分类器。Phase 0 输出的 `cubes[]` 不含 `tactical_type` 字段。如需在特殊场景下启用启发式分类，可修改 `pipeline.py` 在 Step 3 与 Step 4 之间插入分类步骤。

#### 4.3.1 保留的纯几何分析

Phase 0 仍执行以下不依赖语义分类的分析：

**楼板面检测与楼层推断**（§4.2 步骤 4）：`classify_fragments` 将碎片分为 wall / horizontal_plane / other。horizontal_plane 碎片经 `extract_floor_planes`(tolerance=1.0m) 提取楼板面候选，`_merge_adjacent_candidates`(threshold=6.0m) 二级合并为逻辑楼板，`_build_stories_from_slabs` 按 N−1 逻辑生成可居住楼层。典型流程：8候选→4楼板→3可居住楼层。

**墙体线与建筑特征检测**（§4.2 步骤 5）：`extract_wall_lines` 提取 X/Y 向墙体线（66+46）。`detect_rooms_by_story` 按楼层 Z 范围过滤墙体后运行网格房间检测（≥3面围合）。`detect_corridor_candidates` 检测平行墙间隙中长条形低密度矩形（宽1-5m、长≥3m、密度<1.0）。`detect_stairwells_by_density` 在相邻楼层间找紧凑 XY 聚类（1-15m²）+ 高垂直密度（≥0.3）的楼梯井候选。

**几何模式检测**（不按类型分组，纯位置驱动）：
- **等距线性排列**：沿 X/Y/Z 轴等距分布的 prim 序列（如窗格柱、栏杆竖杆）
- **对称对**：关于某轴镜像对称的 prim 对（如门两侧的窗）
- **密集簇**：空间上紧密聚集的 prim 群（如堆积的家具碎片）

模式检测使用 3D 空间哈希优化，O(n) 复杂度，支持 31620 个 prim 全量检测不采样。

**楼梯双方法检测**：(1) `_detect_staircases` 轻量级启发式（step_height 0.08–0.35m, overlap≥0.15, consecutive≥3），(2) `detect_stairwells_by_density` 跨层密度法。两者结果均写入 `patterns`（`tentative_stairs` + `density_stairs`），供 Phase 1 LLM 交叉验证。

## 五、Phase 1：全局场景理解（LLM）

### 5.1 输入与数据精简

```python
def run_phase1(
    scene_metadata_path: str,
    client: Optional[MiniMaxClient] = None,
    output_dir: Optional[str] = None,
    task: Optional[str] = None,
) -> str:
```

Phase 0 输出的 `scene_metadata.json`（31,620 个 Cube，无语义标签）经 `data_compactor` 模块精简后作为 LLM 输入。精简策略如下：

**楼层分组**：

Phase 1 使用 Phase 0 的楼板面证据进行楼层分组。楼层检测基于**楼板面几何证据 + N-1 楼层逻辑**：

- **水平面碎片分类**：`classify_fragments` 按几何特征将碎片分为 wall / horizontal_plane / other
- **楼板面候选提取**：`extract_floor_planes` 从 horizontal_plane 碎片中提取楼板面候选（tolerance=1.0m，可配置）
- **二级合并**：`_merge_adjacent_candidates` (threshold=6.0m) 将相邻候选面合并为逻辑楼板 (slabs)
- **N-1 楼层逻辑**：`_build_stories_from_slabs` 将 slab 之间的空间作为可居住楼层 (stories)。典型流程：8 候选面 → 4 slabs → 3 stories
- **story_count、stories、logical_slab_count** 写入 `floor_evidence` 字段，Phase 1 直接消费
- **参数可配置**：`tolerance=1.0m`（同一楼板碎片合并）、`merge_threshold=6.0m`（同层子结构合并）均为可配置参数

**每层统计信息**（`data_compactor` 为每层生成）：

- `size_stats`：体积 min/median/max，宽度/深度/高度范围
- `material_summary`：前 5 种材质的频次分布
- `representative_cubes`：每层 ≤ 12 个代表性 Cube，采样策略为：
  - 2 个体积最大的（代表结构元素：楼板/大墙体）
  - 4 个 XY 平面四象限代表（确保空间全覆盖）
  - 2 个纵轴极端位置（楼层顶部和底部的代表性物体）
  - 2 个体积中位附近的（代表"典型"碎片）
  - 2 个不同材质的（确保材质多样性）
- `spatial_density`：Cube 在 XY 平面的分布模式（均匀分布 / 中心集中 / 多点聚集 / 稀疏簇），使用 4×4 网格统计法
- 楼板面证据(floor_evidence)和模式检测结果全量保留
- 世界包围盒保留
- **任务描述** `task`（可选）：1-3 句自然语言，传达指挥官意图。若提供，LLM prompt 会附加任务上下文，
  引导场景理解聚焦于与任务相关的空间特征（如"解救人质"→ 关注入口数量、掩体位置、可能的伏击点）。
  输出 JSON 中写入 `_task` 字段供下游阶段读取。

### 5.2 LLM Prompt 策略

System Prompt 设定 LLM 角色为场景理解专家（$A_{\text{parse}}$），User Prompt 由 `format_for_llm()` 将精简数据组装为 ~2000 字结构化文本。

**输入提示策略**：由于 Phase 0 输出的 31,620 个 Cube 全部是三角化碎片（无语义标签），Prompt 开头用 5 条自然语言提示教 LLM 如何从几何碎片中推断建筑结构：

- "某高度附近大量共面碎片 → 可能是地面/楼板"
- "某平面位置一条垂直向密集碎片带 → 可能是墙"
- "两片平行碎片带之间的间隙 → 可能是走廊"
- "碎片围绕的空白矩形区域 → 可能是门/窗开口"

**输出质量约束**（v2 新增）：

| 字段 | 最低要求 |
|------|---------|
| `spatial_layouts` | **≥ 8 个**，覆盖所有空间类型（走廊、房间、楼梯井、入口、外部区域） |
| `tactical_features` | **≥ 8 个**，覆盖瓶颈点、掩体位置、视线走廊、盲区、致命漏斗 |
| `overall_description` | **300-500 字**，分段描述：建筑结构与楼层 → 空间布局与连通关系 → 战术地物与风险 → 环境条件 |

**交叉验证**：Phase 0 的双方法楼梯检测结果（启发式 `tentative_stairs` + 密度 `density_stairs`）与 LLM 的楼梯识别结果进行比对，
一致则确认，不一致则标记 `needs_human_review=true` 供人工复核。

任务描述 `task` 的来源（按优先级）：

若任务描述存在，Prompt 明确要求 LLM **不自行编造 task_hint**，而是将总任务目标分解为每个子场景的阶段性目标。

### 6.1 核心原则：你划分的是空间，不是战术

Phase 2 的职责是把大建筑拆成**空间片段**，每个片段有完整的几何信息。
**不要预判具体怎么打**——那是 Phase 4 的工作。Phase 4 会基于知识库和迭代精炼
从多个可行方案中选择最优战术。

划分原则（优先级从高到低）：

1. **空间完整性（最高优先级）**：spatial_bounds 必须确保裁切后的几何数据自包含——包含全部围合结构、门洞两侧门框 + 门外至少 2 米走廊、楼梯井全高度。**宁可边界重叠，不要几何缺失**。

2. **多入口意识**：不要假设只有一个入口方向。建筑的入口可能在地面层（主入口/侧门/落地窗）、二层（阳台门/窗户/可架梯）、三层（屋顶天窗/阳台/可索降）。每个潜在入口必须有对应的子场景。标注该入口的可达性特征（direct/ladder/rope/breach）。

3. **空间与战术解耦**：用 `space_profile` 描述空间的物理特征（shape/enclosure/vertical_position/entries_exits/key_features），用 `suggested_roles` 提供多种战术建议。Phase 4 可以判定"这个房间可以从走廊突入，也可以从阳台索降进入"，然后根据整体任务选择最优方案。

4. **粒度适中**：每个子场景在裁切后应包含 10-60 个 Cube，每轴跨度不超过 25m。

5. **连通性**：相邻子场景通过 door/corridor/stair/window/open_area/wall_breachable 连接。子场景图反映所有可能的连通路径，不限于单一线性推进路线。

### 6.5 LLM 驱动子场景去重

Phase 2 产出的子场景中可能存在战术等价项——例如多个 F0 小房间虽然位于建筑不同位置，但空间形态、围合程度、门的数量/宽度类别、房间尺度高度相似，导致 Phase 4 为它们生成雷同的战术方案。为此，在子场景划分完成后、进入 Phase 3 前，引入一次 LLM 调用对子场景进行战术等价去重。

#### 6.5.1 设计动机

硬编码阈值（面积差 N%、宽度差 N%）既武断又无法真正理解战术等价性。去重判断必须由 LLM 完成——只有 LLM 能从战术维度综合评估两个场景是否"可以用同一套战术模板应对"。

#### 6.5.2 核心思路

**一次 LLM 调用**：输入全部子场景的定义（space_profile、角色、连通关系、描述），让 LLM 按战术等价性分组，每组选一个代表，输出保留的子场景 ID 列表。下游 Phase 3/4 只处理保留的场景。

#### 6.5.3 战术等价判断标准

"战术等价"的唯一标准：**如果为场景 A 写一份 Phase 4 战术方案（包含突入方式、清房流程、火力配置、队形选择），不加任何修改直接用于场景 B 也完全合理，则 A 和 B 战术等价。**

影响等价的关键维度：

| 维度 | 不等价条件 |
|------|-----------|
| 空间形态 (shape) | room vs corridor vs stairwell vs open_area 完全不同 |
| 围合程度 (enclosure) | fully_enclosed vs partially_open vs fully_open |
| 门的数量与宽度类别 | 单门 vs 多门；narrow vs standard vs wide |
| 房间尺度 | 面积量级不同（如 10m² vs 50m²） |
| 垂直位置 | ground vs mid vs top；不同楼层 |
| 连通场景类型 | 连走廊 vs 连楼梯间 vs 连开放区域 |
| 特殊几何特征 | 致命漏斗、栏杆 vs 实体墙、盲角 vs 开阔视线 |

**特别规则**：
- 走廊 (corridor) 和楼梯井 (stairwell) 通常不应被合并——它们的连通拓扑位置几乎总是独特的
- 入口/外场 (entry_point) 场景各有独特的接近路径，不应合并
- 不同楼层的场景通常不等价
- 宁可保守保留，不要激进删除

#### 6.5.4 选择代表规则

每组只保留一个代表。选择优先级：
1. 位于进攻流更上游的（更靠近建筑入口/外场接近点）
2. 空间结构更典型的（更能代表该组的通用特征）
3. sub_scene_id 字母序更小的（作为最终 tie-breaker）

#### 6.5.5 实现

```python
# src/phase2/dedup_scenes.py

def deduplicate_sub_scenes(
    sub_scenes: List[Dict],
    client: MiniMaxClient,
) -> List[Dict]:
    """对子场景列表进行战术等价去重，返回去重后的子场景列表"""
```

LLM 输出格式：
```json
{
  "kept_sub_scene_ids": ["SS_01", "SS_03", ...],
  "groups": [
    {
      "representative": "SS_01",
      "members": ["SS_01", "SS_02"],
      "removed": ["SS_02"],
      "reason": "两者均为F0标准尺寸房间，单门fully_enclosed，空间尺寸和连通特征接近"
    }
  ]
}
```

**防御设计**：若 LLM 调用失败或返回空结果，保留全部子场景（fail-open）。

#### 6.5.6 调用位置

去重在两处调用，确保冗余场景被彻底移除：

1. **Phase 2 runner**（主入口）：LLM 生成子场景后、写入 `sub_scene_definitions.json` 前调用。同时清理 `sub_scene_graph` 的 nodes/edges 和 `entry_options` 中对已删场景的引用。
2. **Phase 3 pipeline**（安全网）：加载 `sub_scene_definitions.json` 后调用，防止 Phase 2 未去重时进入 Phase 3。

## 七、Phase 3：几何裁切 + 简化 + 描述生成（Python + LLM）

### 7.1 几何裁切（Phase 3a）

对每个子场景的 `overlap_bounds`：

1. 重新扫描原始 USDA 文本
2. 筛选世界空间包围盒与 `overlap_bounds` 相交的 prim
3. 只保留相交 prim，其他丢弃
4. 对跨越边界的 prim（如一堵墙穿过子场景边界），计算该 prim 与子场景 bounds 的体积交集比：
   - 交集比 > 30%：完整保留该 prim（确保建筑结构完整）
   - 交集比 ≤ 30%：丢弃

### 7.2 Mesh → Cube 简化与聚合（Phase 3b，增强版）

Phase 3b 的目标从"每个 Mesh 对应一个 Cube"改为**"每个战术相关元素对应一个 Cube"**。流水线分四步执行：

#### 步骤1：分类与初筛

> **注意**：Phase 3b 的分类是独立的启发式分类步骤，与 Phase 0 无关（Phase 0 仅输出纯几何数据）。Phase 3b 使用自己的几何规则（在 `phase3b_simplifier.py` 中实现，复用 `heuristic_classifier.py` 中的 `TacticalType` 枚举）为裁切后的 prim 标注 `tactical_type`。由于子场景的 prim 数量远少于大场景（数百 vs 数万），启发式分类在子场景级别的准确率可接受。

对 Phase 3a 裁切后的每个 Mesh prim：
1. 使用启发式几何规则打 `tactical_type`（wall/door/window/cover/pillar/floor/stairs/terrain/unknown/decor），分类参数见 `config.py` 中 `PHASE0_HEURISTIC_PARAMS`（Phase 0 启发式分类器已弃用，此参数仅供 Phase 3b 内部使用）
2. 标记为 `architectural_detail` 或 `decorative_cluster` 的 → **直接丢弃**
3. 体积 < 0.005m³ 且 `tactical_type` 非门/窗 → **丢弃**
4. 计算轴对齐包围盒，生成候选 Cube

#### 步骤2：同类型合并

对相同 `tactical_type` 的候选 Cube，满足以下**全部条件**则合并为一个包围盒（聚合参数见 `config.py` 中 `PHASE3B_AGGREGATION_PARAMS`）：
- 质心距离 < `centroid_distance_threshold_ratio` × 场景跨度
- **间隙保护**：两 Cube 之间最短表面距离 < 0.5m（`max_merge_gap = 0.5`）。若间隙 ≥ 0.5m，说明两物体之间存在有意义的空间（可能是门洞/窗洞/通道），不应合并
- `source_material` 相同（`same_material_required = True`）
- 合并后总体积增加 < 30%（`volume_increase_max = 0.30`，防止把分散的独立物体错误合并）

合并后 Cube 的 `size` 和 `translate` 取所有子 Cube 的整体包围盒。

#### 步骤3：模式折叠

对合并后的 Cube 集合执行 §4.3.2 的模式检测：
- **等距线性排列** → 仅保留首尾两个 Cube + 添加 `custom int repeat_count = N`（例如 10 根窗格柱 → 2 根 + `repeat_count = 10`）
- **密集簇**（5+ 个同材质 Cube 在 1m³ 内）→ 合并为单个 `decorative_cluster`，标记为过滤，直接丢弃

此步骤可在实际数据中减少 30-50% 的冗余 Cube（窗格柱群、栏杆竖杆群等）。

#### 步骤4：坐标归一化

将所有 Cube 坐标从世界空间坐标转换为子场景相对坐标：
- 原点 = `(bounds.x.min, bounds.y.min, bounds.z.min)`
- 所有 `xformOp:translate` 的 X/Y/Z 分量各减去原点偏移
- 在根 Xform 上添加 `custom double3 world_origin = (x0, y0, z0)` 保留世界坐标引用

归一化后 LLM 读到的坐标从 `(1350.5, 198.2, 5.5)` 变为 `(0.5, 3.2, 1.5)`，便于直观比较相对位置。

#### 目标指标与输出类型

- 每个子场景（15×15m 典型区域）产出 **≤ 60 个 Cube**（`config.PHASE3B_AGGREGATION_PARAMS["max_cubes_target"]`）
- 若 Cube 数超过 `downgrade_cube_threshold = 100`，自动放宽聚合阈值（`downgrade_distance_multiplier = 1.5`，`downgrade_volume_increase_max = 0.50`）
- 其中标记为 `architectural_detail` 的 Cube 数 **= 0**（已在步骤1过滤）
- `tactical_type` 分类准确率目标 ≥ 80%（Phase 3c LLM 做最终纠正）

输出 Cube 的 `tactical_type` 枚举值精简为 10 类：

| tactical_type | 战术含义 | 对 LLM 生成战术的意义 |
|--------------|---------|----------------------|
| `wall` | 不可穿越的墙体（含楼板/天花板） | 定义空间边界、移动约束、火力扇区限制 |
| `door` | 可穿越的门洞 | 入口/出口候选、突入点 |
| `window` | 可观察但不可穿越的窗 | 观察点、火力点 |
| `cover` | 可提供掩护的独立物体 | 掩体位置、跃进支撑点 |
| `pillar` | 柱状结构 | 掩体、部分遮挡 |
| `railing` | 栏杆/扶手（半高障碍） | 可观察可射击但限制通过 |
| `floor` | 地面/楼板 | 可行走区域 |
| `stairs` | 楼梯 | 垂直通道 |
| `terrain` | 外场地形 | 非建筑区域 |
| `unknown` | LLM 待判定 | 交给 Phase 3c 语义标注时判定 |

以上 `tactical_type` 枚举值与 §4.3.1 启发式分类标签的对应关系为：`wall`←墙体, `floor`←楼板/天花板, `window`←玻璃/窗, `door`←门, `cover`←大型掩体, `pillar`←柱状结构, `terrain`←外场地形, `unknown`←未知结构。`railing` 和 `stairs` 没有对应的启发式规则，由 Phase 1 或 Phase 3c LLM 基于空间上下文判定。`architectural_detail` 和 `decorative_cluster` 不出现在最终输出中（已在 Phase 3b 过滤）。

**注意**：简化阶段保留 `source_material` 字符串（不保留颜色），用于 Phase 3c LLM 辅助推断物体功能。例如 `_Translucent_Glass_Gray_1` → LLM 确认是玻璃窗，`_CorrogateShiny_2` → LLM 确认是波纹金属板（掩体）。

### 7.3 LLM 场景语义标注（Phase 3c，**强制执行**）

Phase 3c 是整个子场景提取流水线的**语义核心**——它将 Phase 3b 产出的归一化 Cube 列表转化为 LLM（Phase 4）可直接理解的结构化场景描述。

**不可跳过**：没有 Phase 3c 的输出，Phase 4 M1 收到的只是坐标列表，无法进行有意义的战术生成。Phase 3c 是连接"几何坐标"和"战术语义"的唯一桥梁。

#### 设计动机

Phase 3b 产出的是几何精确但语义贫乏的 Cube 集合。LLM 虽然能从坐标推断空间关系，但存在三个瓶颈：

1. **坐标算术负担**：LLM 不擅长逐个比较数十个 Cube 的坐标来推断"这两面墙是否相交""这个门通往哪个房间"
2. **分类噪声残留**：Phase 0 启发式分类准确率约 70-80%，仍有 ~20% 误分类需要纠正
3. **战术语义缺失**：坐标无法表达"转角盲区""侧翼暴露""关键控制点"等战术概念

Phase 3c 的核心思想是：**让 LLM 做它擅长的事（语义理解、模式识别、自然语言生成），而非让 LLM 做坐标算术。**

#### 输入

- Phase 3b 输出的归一化简化 USDA 文本（≤60 个 Cube，相对坐标）
- Phase 2 的 `primary_role`（建议战术角色）和 `task_hint`
- **全局上下文**：该子场景在建筑中的位置（楼层、空间形态、围合度、所属 zones、连通关系）
- **相邻子场景摘要**：连通的其他子场景的空间类型、出入口位置、关键威胁
- Phase 1 的全局场景理解（`global_understanding.json`）：提供建筑整体结构、楼层连接关系（楼梯位置）、主要出入口列表等全局上下文

#### LLM Prompt 模板

```
你是一个战术场景分析专家。你将收到一个建筑子场景的简化三维模型。
模型中的每个物体用一个轴对齐立方体(Cube)表示，包含以下属性：
- xformOp:translate：位置（相对坐标，原点=子场景左下角，单位=米）
- size：长宽高（米）
- tactical_type：语义类型（wall/door/window/cover/pillar/railing/floor/stairs/terrain/unknown）
- source_material：原始材质名（如 "_Translucent_Glass_Gray_1"，辅助推断物体功能）

请完成以下分析步骤，输出结构化JSON：

**步骤1：空间分区** — 根据墙体布局，将场景划分为1-5个战术区域(zone)。
   每个zone定义其边界(相对坐标)、空间类型、包含哪些Cube(通过cube_id引用)。
   > **Python 预计算**：zone 的几何边界（连通分量）可由 Python 通过栅格化+BFS 自动标识（将 wall Cube 视为障碍物），LLM 接收预计算的候选 zone 边界，负责：(1) zone 战术类型命名，(2) zone 描述文本生成，(3) zone 间拓扑关系标注。此设计遵循"让 LLM 做语义推理而非坐标算术"的原则。

**步骤2：开口分析** — 识别所有门和窗。
   对每个开口标注它连接的两个zone（或zone与外部"EXTERIOR"）。
   以 Phase 1 全局开口列表为基准（来自 Phase 2 的 `connected_openings`），
   在此基础上基于简化 USDA 中的 Cube（特别是 tactical_type=door/window 的 Cube）做补充和纠错，
   而非从零开始重新扫描。

**步骤3：掩体评估** — 对每个类型为cover/pillar的Cube评估掩护质量：
   - standing：高度>1.5m → 立姿掩体
   - crouching：高度0.7-1.5m → 蹲姿掩体
   - concealment_only：仅提供隐蔽不提供防护（如布料、薄板）
   - obstacle：阻挡移动路线的物体
   同时标注掩体可掩护的方向（north/south/east/west）。
   > **Python 预计算**：`quality` 等级已由 Python 根据 Cube 高度自动预标注（阈值见上述规则）；`coverage_direction` 由 Python 根据 cover Cube 与 wall Cube 的方位关系做启发式推断。LLM 在此基础上验证并修正异常判定，而非从零评估。

**步骤4：威胁推理** — 从空间布局推断潜在威胁（强制输出，至少1条）：
   - blind_corner：转角/门后视线盲区
   - flank_exposure：侧翼暴露位置
   - ambush_position：敌方可能的伏击位置
   - choke_point：瓶颈/窒息点（必须通过且无法展开的狭窄通道）
   - vertical_threat：来自上方的威胁（如楼梯井、中庭）
   - long_sightline：长距离视线的狙击/火力威胁

**步骤5：移动分析** — 分析在该场景内的移动选项：
   - 可用移动路径（贴墙线、掩护跃进线、匍匐线等）
   - 障碍物和物理约束
   - 关键控制点（可控制多个方向/zone的战术要点，一旦控制即可主导场景）

**步骤6：场景摘要** — 100-200字自然语言空间描述。
   要求：一个没看过模型的人读完能画出示意图。
   描述空间形状、墙/门/窗的相对位置、主要物体布局。

**步骤7：标签生成** — 3-8个自由形式场景标签。
   如"L形转角""多门走廊""阳台暴露""掩体富裕"等。
```

- 若 `tactical_type = unknown` 的 Cube 可以被判定 → 在 `cover_assessment` 中引用并给出判定
- 纠正不需要修改 USDA 文件，仅在 desc.json 中以文字描述纠正即可

```
    └── ...
```

**Phase 4 M1 的实际输入**以 `desc.json` 为主（语义层）、`scene.usda` 为辅（几何参考）。M1 不再需要从坐标重新推断空间关系——Phase 3c 已在 `desc.json` 中完成了空间分区和拓扑标注。

> **Phase 3c 输出后的自动校验（Python 脚本执行）**：在 Phase 3c LLM 语义标注完成、desc.json 写入后，Python 脚本自动执行以下坐标一致性检查（不做 LLM 依赖的语义判断）：
> 1. 检查每个 zone 的 `contained_cube_ids` 对应的 Cube 坐标（从 scene.usda 读取）是否落在该 zone 的 `bounds_rel` 范围内，偏差超过 0.5m 的报告警告
> 2. 检查 `cover_assessment` 中的 `cube_id` 是否在 scene.usda 中存在
> 3. 检查 `openings` 中引用的 zone_id 是否在 `zones` 列表中
> 4. 统计 scene.usda 中 `tactical_type = door/window` 的 Cube 数量与 desc.json `openings` 条目数的差异，差异超过 2 的报告警告
> 不通过校验的子场景标记为 `validation_failed`，不进入 Phase 4。

### 7.5 LLM-Consumable 子场景格式规范（新增）

本节解释为什么经过 Phase 3 完整处理后，子场景格式对 LLM 空间推理友好，以及格式设计的核心原则。

#### 7.5.1 核心原则：让LLM做语义推理而非坐标算术

LLM 在空间推理方面的强项和弱项：

| LLM 擅长 | LLM 不擅长 |
|---------|-----------|
| 理解"走廊两侧对称分布4个门" | 从 32 个 (x,y,z) 坐标推断哪 4 个 Cube 构成门 |
| 评估"转角存在盲区"的威胁 | 计算两个 (x1,y1) 和 (x2,y2) 是否形成转角 |
| 规划"利用桌子和墙角分段跃进" | 判断 (1350.3, 195.9, 7.0) 和 (1355.6, 195.9, 7.0) 是否是同一面墙 |
| 从材质名推断物体功能 | 从尺寸判断 0.4×0.02×0.9m 是装饰条还是栏杆 |

因此，子场景格式的设计原则是：**Phase 3 做完所有坐标推理，输出语义结构。Phase 4 在语义结构上做战术推理。**

#### 7.5.2 格式要素对比：旧 vs 新

| 维度 | 旧格式（仅 Phase 3b 输出） | 新格式（Phase 3c 完整输出） |
|------|--------------------------|--------------------------|
| **LLM 要理解的内容** | 200+ 个 Cube 的坐标和 tactical_type | ~5 个 zone + 拓扑关系图 + ≤60 个 Cube |
| **空间推理方式** | 对每个 Cube 做坐标比较推断邻接关系 | 读 `zones[].connected_to` 直接获取邻接关系 |
| **开口理解** | 从 Cube 坐标和 wall Cube 推断"门在哪个墙上" | 读 `openings[].connects` 直接知道门连接哪两个 zone |
| **掩体判断** | LLM 从 size 自己算掩护高度和质量 | 已预标注 `quality`（standing/crouching）和 `coverage_direction` |
| **威胁分析** | LLM 完全从头推理 | 已有 `inferred_threats` 作为推理起点（可补充不可依赖） |
| **坐标值域** | 世界坐标 1350–1410（数字大、无直观意义） | 相对坐标 0–15（数字小、可直接比较相对位置） |
| **Cube 数量** | 200+ | ≤60（过滤+合并+模式折叠后） |

#### 7.5.3 战术生成可用性自检清单

每个子场景在进入 Phase 4 前，应通过以下自检（可由 Python 脚本自动执行 + Phase 3c LLM 自检补充）。

**Python 脚本自动检查项**（§7.4 的坐标校验已覆盖）：
- [ ] `scene.usda` 中 Cube 数量 ≤ 60
- [ ] 所有 `xformOp:translate` 分量在 `[0, sub_scene_span]` 范围内（坐标已归一化）
- [ ] 无 `tactical_type = architectural_detail` 或 `decorative_cluster` 的 Cube 残留
- [ ] `cover_assessment` 中的 `cube_id` 均在 scene.usda 中存在
- [ ] `openings` 中引用的 `zone_id` 均在 `zones` 列表中存在

**语义内容检查项**（与 §7.3 强制约束表一致，由 Phase 3c LLM 或 M1 验证）：
- [ ] `zones` 数量 ≥ 1（§7.3 强制约束）
- [ ] `inferred_threats` 非空（§7.3 强制约束）
- [ ] `movement_analysis.key_control_points` 非空（§7.3 强制约束）
- [ ] `spatial_description` 在 100–200 字范围内（§7.3 强制约束）
- [ ] `inferred_tags` 在 3–8 个范围内（§7.3 强制约束）

未通过自检的子场景**不应进入 Phase 4**，应先修复 Phase 3 的问题。

### 8.3 M2：自适应生成策略

M2 决定每个子场景的战术生成模式。采用双层决策机制：

#### 第一层：检索置信度

将场景描述文本与 PDF 参考资料各章节进行语义相似度计算：

$$\text{sim}(s, ch_i) = \text{Embed}(\text{desc}(s)) \cdot \text{Embed}(ch_i)$$

取最高相似度：

$$\text{sim}_{\max}(s) = \max_{i} \text{sim}(s, ch_i)$$

判定规则：
- 若 $\text{sim}_{\max}(s) < \theta_{\text{low}}$：$m(s) = GEN$（直接进入自主生成，**跳过第二层 LLM**）
- 若 $\text{sim}_{\max}(s) > \theta_{\text{high}}$：$m(s) = RAG$（直接进入检索增强生成，**跳过第二层 LLM**）
- 若 $\theta_{\text{low}} \leq \text{sim}_{\max}(s) \leq \theta_{\text{high}}$：进入第二层 LLM 自评判断

其中 $\theta_{\text{low}}, \theta_{\text{high}}$ 为预设阈值，具体取值通过实验标定。第二层 LLM 自评仅在嵌入相似度处于"灰色地带"时触发；设灰色地带占比为 $P_{\text{mid}}$，则平均每个子场景可节省 $(1 - P_{\text{mid}})$ 次第二层 LLM 调用。

#### 第二层：LLM 自评

将检索到的参考内容 $K(s)$ 交由 LLM 评估其对当前场景的适用性：

$$\text{Assess}(s, K(s)) \rightarrow \{\text{适用}, \text{部分适用}, \text{不适用}\}$$

- **适用**：$m(s) = RAG$
- **部分适用**：$m(s) = Hybrid$，LLM 同时输出 $\{K_{\text{usable}}, K_{\text{insufficient}}\}$，标注哪些部分可参考、哪些需要自主生成
- **不适用**：$m(s) = GEN$

#### 三种生成模式

| 模式 | 符号 | 输入 | 说明 |
|------|------|------|------|
| 检索增强生成 | $G_{\text{RAG}}$ | 场景 + 检索到的PDF章节 | 以已有战术知识为主要参考 |
| 混合生成 | $G_{\text{Hybrid}}$ | 场景 + 部分PDF章节 + LLM自主知识 | 参考与自主相结合 |
| 自主生成 | $G_{\text{GEN}}$ | 场景（无参考） | 完全依赖LLM的军事知识 |

#### 知识检索策略

当 $m(s) \in \{RAG, Hybrid\}$ 时，执行**章节级语义检索**：

1. **索引构建**：将 PDF 参考资料按章节拆分，对每个章节 $ch_i$ 计算嵌入向量 $\text{Embed}(ch_i)$ 并建立向量索引
2. **检索执行**：以场景描述 $\text{desc}(s)$ 为查询，检索 Top-$k$ 相关章节（建议 $k=3$）
3. **上下文组装**：将检索到的章节内容拼接为生成时的参考上下文

> **检索源扩展计划**：当前检索源为 $\mathcal{R}$（PDF 参考资料）。随着知识库 $\mathcal{K}$ 中积累足够数量的高质量已验证战术（H 级），M2 的检索范围可扩展为 $\mathcal{K} \cup \mathcal{R}$ 双源检索——既检索 PDF 教范知识（权威原则），也检索知识库中相似场景的已验证战术（经验复用）。双源检索的融合策略（加权拼接/分别检索取并集/知识库优先）待实验确定。

#### 工程实现细节

M2 的实际实现包含以下关键设计：

1. **嵌入客户端降级链**：`本地 BGE → MiniMax → OpenAI → stub 零向量`。本地 BGE 模型从 `data/models/bge-large-zh-v1.5/` 加载（可通过 `BGE_LOCAL_PATH` 环境变量覆盖），失败后依次尝试云端 API，最后降级为零向量（M2 退化为 GEN 模式）。
2. **RETRIEVAL 容错**：`embed()` 和 `vector_store.search()` 包裹在 try/except 中，任何环节失败均回退到 GEN 模式，不阻断流水线。
3. **零向量检测**：若嵌入结果全为零（stub 降级），直接返回 GEN 模式。
4. **参考资料聚合**：仅检索与场景相似度 ≥ $\theta_{\text{low}}$（0.3）的章节，拼接为参考上下文注入后续阶段。

### 8.3b Stage 0：穷举枚举生成（新增）

M2 完成后，传统做法是直接进入 M3 生成**单条**战术。但单一战术无法覆盖场景的所有可能性——不同的突入角度、编队配置、节奏策略可能都适用于同一场景。Stage 0 穷举枚举机制通过**参考资料分批注入 + LLM 迭代发现**，在 M3 之前生成多个不重复的战术概念。

#### 核心机制

```
参考资料 (可能很长) ──► 按段落分批 (2000字/批)

For 每批参考 + 每轮无参考:
  ┌──────────────────────────────────────────────┐
  │  LLM 接收:                                    │
  │    1. 场景 desc.json                          │
  │    2. 本批参考资料 (或 "无更多参考")            │
  │    3. 已生成战术列表 (含名称/目标/动作链)       │
  │                                              │
  │  LLM 输出:                                    │
  │    - 新战术概念 JSON 数组 (每个含核心字段)      │
  │    - 已穷尽时返回 []                          │
  └──────────────────────────────────────────────┘
           │
           ▼
  Python 层去重:
    ├─ 名称为空 → 丢弃
    ├─ Tactic_Name 完全匹配 → 丢弃
    ├─ 字符二元组 Jaccard > 0.8 → 丢弃
    └─ 通过 → 加入集合
```

#### 去重算法：字符二元组 Jaccard

由于战术描述为中文文本，传统空格分词方法无法产生有意义的词。采用字符二元组（character bigram）方法：

$$\text{Jaccard}(t_1, t_2) = \frac{|B(t_1) \cap B(t_2)|}{|B(t_1) \cup B(t_2)|}$$

其中 $B(t)$ 提取文本中相邻两字符构成的二元组集合（仅保留中英文字符，跳过标点和空格）。

#### 终止条件

| 条件 | 说明 |
|------|------|
| 达到 15 个战术 | 硬上限，防止过度生成 |
| 连续 2 轮无新产出 | 仅在无参考阶段计数——有参考时本轮无产出不终止 |
| 绝对轮次上限 | `总批次数 + 无参考最大轮数 + 2`，防止无限循环 |

#### 与 M3 的衔接

穷举阶段产出的是**轻量级战术概念**（名称、目标、描述、动作链摘要、语义标签），不含完整的 text_version + struct_version 双版本结构。每个概念作为 `seed_concept` 参数传入 M3，A_gen 负责将其扩展为符合全部 19 条通用性规则的双版本完整战术。

### 8.4 正向 Few-Shot 示例引导系统

> **前置说明**：本节讨论的 Few-Shot 示例系统是 M3 多智能体协作框架（§8.5）的增强组件——示例注入 $A_{\text{gen}}$ 的 system prompt，在 M3 迭代精炼开始前生效。M3 的完整定义（Agent 角色、迭代协议、终止条件、审查规则）见 §8.5。

#### 8.4.1 设计动机

当前 M3 的通用性保障依赖**负向约束**——通过 G-T1~G-T12、G-S1~G-S6 规则告诉 LLM"禁止写什么"，由 $A_{\text{review}}$ 事后检查违规。这种方式存在三个固有局限：

1. **缺乏正向示范**：LLM 知道不能写数字、不能写具体物体名，但不知道在那些位置"应该写什么"。禁掉了"3米"，LLM 可能写出"约三米处"（G-T2 捕获）或"几步距离"（所有规则漏检）。
2. **审查-修正循环消耗 token**：每轮审查发现违规 → 修正 → 再审查，迭代成本高。如果 LLM 在首次生成时就掌握泛化模式，可以压缩迭代轮次。
3. **GEN 模式缺乏锚定**：当 M2 判定为 GEN 模式（无 PDF 参考）时，LLM 完全依赖自身知识生成，质量波动最大。固定示例集为 GEN 模式提供最低限度的格式锚定。

正向 Few-Shot 示例系统的核心思想是：**用对比示例让 LLM 在生成前就学会"泛化模式"，而非在生成后用规则纠正。** 定位为通用性审查规则（§8.5.4）的补充——负向规则负责"拦"，正向示例负责"教"。

#### 8.4.2 示例格式：对比对

每条示例是一个**对比对**，包含：违规版本（常见错误写法）、修正版本（通用化写法）、转换标注（解释改了什么）。


每个对比对包含三个要素：
| 要素 | 说明 |
|------|------|

| **违规版/修正版并排** | 同一条战术意图的两种写法，LLM 通过对比直接看到"改了什么" |
| **转换对照表** | 逐项标注每个违规词对应的修正词和转换类型（角色化/空间泛化/时间泛化/火力泛化/物体泛化/方位泛化） |

#### 8.4.3 固定示例集设计

系统维护一个固定示例集（7 条），覆盖最核心的通用性转换模式。**所有子场景共用同一套示例**，不做场景匹配和动态选择。

**设计原则**：
- **覆盖违规类型**：每条示例包含 3 种以上的违规类型，确保常见违规模式都有演示
- **覆盖战术阶段**：示例分散在侦察、进攻、防御、撤离四个阶段
- **覆盖空间类型**：开阔地、走廊、房间、楼梯各至少一条
- **文字描述版优先**：示例以自由文本 Intent 字段为主要对象（结构化描述版的占位符替换模式较机械，不如文字描述版的泛化模式复杂）

**推荐示例集（7 条）**：

| 示例ID | 场景类型 | 战术阶段 | 主要演示的转换类型 |
|--------|---------|---------|------------------|
| EX-01 | 走廊转角 | 进攻—肃清 | 角色化、方位泛化、空间泛化、火力泛化 |
| EX-02 | 开阔地接近 | 进攻—接近 | 物体泛化、距离泛化、时间泛化 |
| EX-03 | 房间突入 | 进攻—清剿 | 角色化、空间泛化、战术动作泛化 |
| EX-04 | 楼梯推进 | 进攻—肃清 | 方位泛化（高低位）、火力泛化、协同时序泛化 |
| EX-05 | 撤退掩护 | 撤离 | 角色化、物体泛化、距离泛化、时间泛化 |
| EX-06 | 房间肃清 | 进攻—肃清 | 物体泛化（家具→掩体）、方位泛化、角色化、空间泛化 |
| EX-07 | 走廊肃清 | 进攻—肃清 | 编队规模泛化（编队人数→编组交互）、角色化 |

> **为什么用固定集而非场景匹配**：固定集的设计选择基于以下考量：(1) Few-shot 示例的本质作用是教 LLM **转换模式**而非**场景具体做法**——转换模式是跨场景通用的；(2) 固定集避免了匹配不准引入噪声的风险；(3) 7 条示例的总 token 量约 4,000-6,000 token，在 LLM 上下文中占比可控，无需精简。若未来示例量扩充至 20+ 条，可重新评估匹配机制的必要性。

#### 8.4.4 Prompt 集成

固定示例集注入 $A_{\text{gen}}$ 的 system prompt，在**首次生成 $T^0$ 之前**即生效（不是修正阶段才注入）。

注入量按生成模式调整：

| 生成模式 | 注入示例数 | 理由 |
|---------|-----------|------|
| RAG | 4 条（固定选 EX-01, EX-03, EX-06, EX-07） | PDF 参考资料已提供格式锚定，少量示例即可补充泛化模式 |
| Hybrid | 5 条（固定选 EX-01, EX-02, EX-03, EX-06, EX-07） | PDF 部分参考，示例补足自主生成部分的格式约束 |
| GEN | 全文 7 条 | 无参考资料，few-shot 是唯一格式引导来源 |

注入位置（$A_{\text{gen}}$ system prompt 结构）：

```
你是一个战术生成专家。请基于场景理解生成通用化战术方案。

=== 格式要求 ===
[现有的角色化原则、场景无关性原则、相对关系原则描述 — 保持不变]

=== 正向示例：正确的通用化写法 ===
以下示例展示"过于具体的违规写法"与"通用化修正写法"的对比。
请你学习修正版中的泛化模式，在生成战术时应用相同的转换逻辑。

[注入 k 条示例，每条为场景上下文 + 违规版 + 修正版 + 转换对照表]

=== 场景输入 ===
[当前子场景的 M1 场景理解 JSON]
```

关键设计细节：
- 示例放在"场景输入"**之前**——LLM 先学习转换模式，再处理具体场景，减少先入为主的倾向
- 保留现有原则描述（角色化/场景无关性/相对关系）作为规则框架，示例作为规则的具体演示
- 不要求 LLM 复制示例的战术内容，只学习转换模式（角色化、泛化、去数值化）

#### 8.4.5 示例库的演进与维护

**初始构建**（Phase 4 开发阶段）：
- 人工编写 5 条初始示例，覆盖最关键的五种场景类型
- 每条示例经 $A_{\text{review}}$ 验证：违规版能触发至少 3 条通用性规则的违规，修正版不触发任何违规

**持续扩充**（知识库运行阶段）：
- **从审查通过的战术中提取**：将 M4 评分 ≥ 8.0 的优质战术的 Intent 字段与 $A_{\text{gen}}$ 的初稿对比，提炼出"LLM 自主完成的泛化转换"，作为正向示例加入
- **从高频违规中生成**：统计 $A_{\text{review}}$ 反馈中高频出现的违规类型，若某类违规在已有示例中未被充分覆盖，人工编写新示例并加入

**版本控制**：
- 示例库与战术模板同步版本管理
- 每次通用性审查规则（G-T1~G-T12, G-S1~G-S6）更新时，检查现有示例是否仍正确覆盖新规则

#### 8.4.5 核心任务认知框架（A_gen Prompt 增强）

除了 Few-Shot 正向示例（§8.4.1-§8.4.4），A_gen 的 system prompt 在 Role 定义之后、设计原则之前增加了**"核心任务：战术模式抽象"**段落。该段落的目的是在 LLM 开始生成**之前**建立正确的任务认知——让 LLM 理解它做的是**战术模式抽象**，而非**场景脚本编写**。

核心任务段落包含三个关键要素：

1. **任务本质重定义**：明确告知 LLM，其根本任务不是"为当前场景编写一份行动脚本"，而是"从当前场景中抽象出可复用的战术行动模式"。通用化战术被比喻为"野战手册的条目"——适用于任何具备相同空间拓扑类型的场景。

2. **自检判断标准**：提供 LLM 可自主执行的通用性验证——"如果把战术中所有对具体地点的指代全部去掉，替换为空间类型描述（'走廊区域'、'转角位置'、'入口门廊'等），战术是否依然完整、正确、有意义？"如果是，说明写的是通用战术；如果不是，说明写的是场景脚本。

3. **desc.json 角色明确定义**：明确 desc.json 是"用于理解空间拓扑关系的输入数据"，而非"战术输出的名词来源"。显式指出 `zone_id`、`opening_id`、`sub_scene_id` 等标识符是场景标注信息，**不是**让 LLM 写进战术输出的。战术输出中，空间位置用其**空间类型和功能编组**来描述，而非其标注编号。

该段落与现有机制的关系：

| 机制 | 层次 | 作用方式 |
|------|------|---------|
| **核心任务段落** | 认知层 | 在生成前框定任务本质，让 LLM 理解"为什么"要做通用化 |
| **Few-Shot 示例** (§8.4) | 学习层 | 通过对比对展示"怎么"做通用化转换 |
| **设计原则 + 规则清单** (§8.5.4) | 执行层 | 提供具体约束条目作为"检查清单" |
| **G-T11 审查规则** (§8.5.4) | 安全网 | 即使前三层失效，审查仍能捕获英文词汇泄露 |

设计意图：**不在规则层继续堆叠规则，而是在认知层让 LLM 理解任务的本质。** 新增的 G-T11 仅作为安全网——当认知框架、Few-Shot 示例、设计原则三层均未阻止英文词汇泄露时兜底。

### 8.5 M3：多智能体协作框架

M3 对 Stage 0 产出的**每个战术概念**独立执行迭代精炼（逐概念循环）。当 `seed_concept` 参数非空时，A_gen 基于给定概念扩展为完整双版本战术而非从零设计。

#### 8.5.1 Agent 角色定义

| Agent | 符号 | 职责 | 知识来源 | 实现文件 |
|-------|------|------|----------|---------|
| 战术生成Agent | $A_{\text{gen}}$ | 基于场景（+可选种子概念）生成/扩展战术方案（文字描述版+结构化描述版） | LLM + PDF参考/自主知识 | `agen_prompts.py`（按 RAG/Hybrid/GEN 模式注入 Few-Shot 示例，支持 seed_concept 扩展） |
| 军事审查Agent | $A_{\text{review}}$ | 通用性审查（G-T1~G-T12 文字描述版 + G-S1~G-S6 结构化描述版）+ 军事可行性审查 | 结构化规则文件 | `areview_prompts.py` + `review_schema.py`（ReviewFeedback 结构化输出） |
| 质量评估Agent | $A_{\text{eval}}$ | 对战术进行 6 维度评分 | 评价 rubric | `aeval_prompts.py`（A_eval system prompt） |

> **M3 实现架构**：迭代协议由 `M3IterationLoop` 类（`iteration_loop.py`）驱动，A_gen 和 A_review 均通过 `MiniMaxClient.generate_json()` 调用。通用性审查的 19 条规则全部由 A_review LLM 语义执行（不依赖前置正则预检——正则已被废弃，因中文语义判断无法用正则可靠完成）。审查上下文由 `precheck.extract_review_context()` 从战术 JSON 中提取。Few-Shot 示例通过 `few_shot_examples.format_examples_for_prompt(mode)` 按生成模式注入不同数量的示例（RAG=6条, Hybrid=7条, GEN=9条）。

> **注意**：场景解析（空间布局理解、描述生成、标签生成）由前置阶段完成——Phase 1 LLM（全局场景理解，输出格式沿用 "A_parse 格式" 命名，见 §5.3）和 Phase 3c LLM（子场景语义标注，见 §7.3）。M3 框架在迭代精炼阶段介入时，场景解析已是完成的输入，M1（§8.2）在此基础上做一致性验证和战术标注补充。

#### 8.5.2 辩论式迭代精炼协议

采用**生成 → 通用性审查 → 军事审查 → 修正**的辩论式迭代精炼，最大迭代轮次 $N_{\max} = 3$：

```
场景 s + 生成模式 m(s)
        │
        ▼
┌─────────────────────────┐
│  A_gen: 初始生成 T⁰      │  ← 第1轮
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│  A_review: 通用性审查     │
│  检查: G-T1~G-T12         │
│        G-S1~G-S6         │
│  输出: {通过, 违规报告}   │
└────────────┬────────────┘
             │
      ┌──────┴──────┐
      ▼              ▼
   通过            未通过
      │              │
      ▼              ▼
┌─────────────────┐  ┌─────────────────┐
│  A_review: 军事审查│  │  A_gen: 通用性修正│
│  输出: {通过,反馈} │  │  T⁰' = Fix(T⁰)   │
└────────┬────────┘  └────────┬────────┘
         │                    │
         │              重新进入通用性审查(最多2次修正)
         ▼
      ┌──────┴──────┐
      ▼              ▼
   通过            未通过
      │              │
      ▼              ▼
  进入A_eval    ┌─────────────────┐
               │  A_gen: 军事修正  │  ← 第2轮
               │  T¹ = Fix(T⁰', F⁰)│
               └────────┬────────┘
                        ▼
               ┌─────────────────┐
               │  A_review: 再审查 │
               │  (通用性+军事)    │
               └────────┬────────┘
                        │
                 ┌──────┴──────┐
                 ▼              ▼
              通过            未通过
                 │              │
                 ▼              ▼
             进入A_eval    ┌─────────────────┐
                          │  A_gen: 定向修正  │  ← 第3轮
                          │  T² = Fix(T¹, F¹)│
                          └────────┬────────┘
                                   ▼
                          ┌─────────────────┐
                          │  A_review: 终审   │
                          │  (通用性+军事)    │
                          └────────┬────────┘
                                   │
                            ┌──────┴──────┐
                            ▼              ▼
                         通过            未通过
                            │              │
                            ▼              ▼
                        进入A_eval      丢弃该战术
```

**符号约定**：流程图中 $F^k$ 表示第 $k$ 轮军事审查后 $A_{\text{review}}$ 输出的结构化反馈（格式见 §8.5.5），包含违规报告和改进建议。$A_{\text{gen}}$ 在修正时以 $F^k$ 为修正依据。

**早期丢弃机制**：首次军事审查（通用性审查通过后的第一次军事审查）中，若 $A_{\text{review}}$ 判定军事可行性维度评分 $< 3.0$，则跳过修正循环，直接丢弃该战术。理由：评分 $< 3.0$ 对应"多处关键遗漏或逻辑不自洽，需整体重构"（见 §8.6.1 通用评分锚点 2-3 分段定义），此类结构性缺陷无法通过定向修正修复——修正只能解决局部违规，无法挽救整体逻辑崩塌的战术。丢弃后由 $A_{\text{gen}}$ 以不同随机种子或调整后的上下文重新生成 $T^0$，而非在错误框架上迭代。

**审查顺序说明**：通用性审查优先于军事审查。原因：通用性违规（如出现具体数值）是格式性问题，修正成本低；军事审查是内容性问题，应在格式正确后再进行。这样避免军事审查通过后因通用性违规被推翻的无效迭代。

**通用性审查的循环控制**：通用性审查采用"审查→修正→再审查"的循环（不增加军事修正计数），但单轮内最多允许 **2 次通用性修正**（即：初始审查 + 最多 2 次修正 = 最多 3 次通用性审查）。若 2 次修正后通用性审查仍未通过，保留最后一次修正结果进入军事审查（通用性违规在军事审查反馈中标记为残留问题，由后续军事修正一并处理）。此设计避免通用性问题阻碍整体流程推进。

#### 8.5.3 终止条件

| 条件 | 判定标准 | 结果 |
|------|----------|------|
| 通过审查 | $A_{\text{review}}$ 评分 $\geq Q_{\text{pass}}$（审查通过阈值，建议 $Q_{\text{pass}} = 7.0$） | 进入质量评估 |
| 军事不可行 | 首次军事审查中军事可行性维度评分 $< 3.0$（评分锚点：2-3 = "多处关键遗漏或逻辑不自洽，需整体重构"） | 丢弃该战术，不进入修正循环 |
| 达到最大轮次 | 军事审查轮次 $= N_{\max} = 3$（通用性修正不计入轮次，通用性最多单独修正 2 次后强制进入下一阶段） | 取当前最优版本进入评估 |
| 评分收敛 | 连续两轮军事审查评分提升 $< \delta = 0.5$ | 停止迭代，取当前版本 |

> **$Q_{\text{pass}}$ 与质量分级阈值 $Q_{\text{H/M/L}}$ 的关系**：$Q_{\text{pass}} = 7.0$ 是 M3 迭代终止的"及格线"（审查通过即可结束修正），而 M4 的入库分级阈值 $Q_{\text{H/M/L}}$（H ≥ 8.0, M ≥ 6.0, L < 6.0）决定战术的最终质量标签。一个通过 M3 审查的战术（评分 ≥ 7.0）在 M4 完整评价后可能被标记为 M 级或 H 级。$Q_{\text{pass}}$ 的取值略高于 M 级下限（6.0），确保入库战术至少达到 M 级标准。

#### 8.5.4 通用性审查规则

> **实现说明**：当前实现中，全部 19 条通用性规则（G-T1~G-T12, G-S1~G-S6）由 $A_{\text{review}}$ LLM 语义审查（通过 `areview_prompts.py` 中 `build_areview_input()` 构建审查 prompt），**不依赖前置正则预检**。早期版本的正则匹配方案已被废弃——中文文本的语义判断（区分方向用字"一侧"与数量用字"一名"、区分步骤编号"步骤1"与数值"1米"）无法用正则可靠完成。$A_{\text{review}}$ 对所有规则执行语义级审查并输出结构化 `ReviewFeedback`（`review_schema.py`），区分硬约束违规（触发修正）和软约束违规（记录警告）。

以下规则表保留规则定义和正则模式，供参考和理解每条规则的意图。

### 8.6 M4：质量评价与入库

**M3 审查评分与 M4 质量评分的区别**：两者使用同一套六维度 Rubric（§8.6.1-§8.6.7），但服务于不同目的：

| 对比维度 | M3 审查评分 | M4 质量评价 |
|----------|-----------|-----------|
| **目的** | 迭代终止条件（是否继续修正） | 入库分级依据（H/M/L） |
| **评分者** | $A_{\text{review}}$（审查Agent） | $A_{\text{eval}}$（评估Agent） |
| **时机** | 每轮迭代后立即评分 | 迭代终止后最终评价 |
| **使用方式** | 全维度评分，但以维度四（通用性合规）和维度六（军事可行性）为主要迭代判据——通用性违规触发修正、军事缺陷触发重新生成；其余四维作为辅助参考 | 六维度等权综合评分，决定知识库中的质量标签 |
| **关注重点** | 可阻断迭代的致命问题（硬约束违规、军事原则错误） | 全维度综合卓越性 |

M4 的评价结果（而非 M3 最后一次评分）作为战术入库时的 `Credibility` 字段值和 `Quality_Level` 标签。

#### 8.6.1 通用评分锚点

六个维度均采用统一的 0-10 分制。以下锚点定义适用于全部六个维度，确保不同维度间的评分可比：

| 分数 | 等级标签 | 锚点定义 |
|------|---------|---------|
| **10** | 卓越 (Exemplary) | 表现超出条令预期，主动考虑了边界情况和后备方案（Branch Plan），可作为其他战术生成的正面范例（Add to Few-Shot Pool） |
| **8-9** | 优秀 (Proficient) | 完全达标，无遗漏，关键判断有可追溯的逻辑链条，无需人工干预可直接使用 |
| **6-7** | 合格 (Satisfactory) | 基本达标，存在少量不影响核心执行的细节瑕疵，可由指挥官微调后使用 |
| **4-5** | 有缺陷 (Marginal) | 存在明显的设计缺陷，当前状态下无法直接执行，需定向修正 |
| **2-3** | 严重缺陷 (Unsatisfactory) | 多处关键遗漏或逻辑不自洽，需整体重构，定向修正不足以修复 |
| **0-1** | 不可接受 (Unacceptable) | 违反基本军事原则或物理规律，执行必然导致任务失败或不可接受的损失 |

> **通用锚点与维度锚点的分段对应**：各维度评分锚点表（§8.6.2-§8.6.7）为简洁起见将底部两段合并为"0-2"一行。评分时，维度锚点的"0-2"应参照上表中 2-3（严重缺陷/需整体重构）与 0-1（不可接受/违反基本原则）的区分——前者尚有重构可能，后者应直接否决。M3 早期丢弃阈值（3.0）和 M4 否决条件 V1-V3（3.0）均依赖此区分。

#### 8.6.2 维度一：场景适配度 (Scene Adaptation)

> 评估战术是否充分回应了场景（desc.json + task_hint）中呈现的空间挑战类型和战术态势。**注意：通用化战术不应点名具体的 zone_id 或 opening_id**（那是场景绑定层的工作），而应以空间关系的模式化语言（"走廊转角""楼梯口""开阔地"）描述各类空间挑战的战术处置方案。

**评分 Checklist（逐项核查）**：

| 编号 | 核查项 | 核查方法 |
|------|--------|---------|
| S1 | **空间挑战类型全覆盖** | desc.json 中可以归纳出哪些空间挑战类型（走廊类/转角类/房间类/楼梯类/开阔地类/门窗口类）？战术是否对每一种出现的类型给出了对应的处置模式？（如"走廊推进"对应了抵近侦察+交替掩护模式，"房间突入"对应了切片侦察+动态突入模式）按类型归纳后交叉匹配，而非逐个 zone_id 点名 |
| S2 | **开口利用模式正确** | 场景中存在的开口（门/窗/阳台口）在战术中是否被正确纳入战术逻辑？——作为信息获取点（侦察口）、火力投射点（压制口）、或物理突破点（突入口）？战术是否体现了"先侦察后通过/先压制后突入"的开口处置原则？关注处置模式的合理性，而非开口是否被逐一列举 |
| S3 | **掩体利用意识** | 场景中可用的掩体类型（沿墙掩体/大型独立掩体/转角掩体/高低位掩体）是否在战术机动描述中得到利用？战术是否体现了"掩体间跃进"、"掩体后方建立火力点"等掩体利用模式？关注掩体利用的战术意识，不要求点名每个具体掩体 |
| S4 | **威胁类型全应对** | inferred_threats 中归纳的威胁类型（如"转角后方伏击风险""远距离火力覆盖风险""门后伏击风险"）是否在战术中均有对应的应对模式？（压制/规避/优先侦察/诱导暴露）按威胁类型归纳后匹配，而非逐个威胁实例点名 |
| S5 | **task_hint 全回应** | task_hint 中的阶段性战术目标是否在 Objective/Description 中有明确呼应？该目标是战术生成的核心驱动，不可遗漏。语义匹配 task_hint → Objective |
| S6 | **环境条件适配** | 战术是否体现了对光照（白天/夜间）、天气、视野条件的适配？（如夜间/视野受限场景是否采用了更谨慎的推进节奏、更近的侦察距离、更强调声音/热成像等替代感知手段）语义检查 |
| S7 | **空间关系正确** | 战术中对空间关系的描述是否与 desc.json 中的空间拓扑兼容？（如"从走廊进入房间"在拓扑上是否可行？拓扑中不相连的 zone 之间是否被错误地安排了直接机动？）拓扑兼容性验证，不要求枚举每个 zone 的连接 |
| S8 | **无场景外假设** | 战术中不应建立在场景中不存在的空间条件之上（如场景无阳台/屋顶通道却出现"从阳台突入""从屋顶索降"；场景为开阔地却出现"沿走廊推进"）。战术对空间类型的判断应与 desc.json 一致。反向核查 |

**评分锚点**：

| 分数 | 判定标准 |
|------|---------|
| **9-10** | S1-S9 全部通过。战术准确识别了场景中的全部空间挑战类型，每种类型均有对应的高质量处置模式。主动挖掘了空间布局中的战术优势（如利用视线关系建立交叉火力）。所有描述以模式化语言表达，无场景特定ID引用。场景不确定性被识别并纳入了稳健策略 |
| **7-8** | S1-S6 全部通过。S7 通过（无拓扑错误）。S8 通过（无场景外假设）。所有空间挑战类型得到回应但模式描述可能存在少量可优化空间（如某类挑战的处置方案正确但不够精细） |
| **5-6** | S1-S5 的核心类型覆盖率达到 80% 以上。存在 1-2 种空间挑战类型未被充分回应（如场景存在"楼梯推进"挑战但战术未给出楼梯特有的处置模式），但不影响战术主体逻辑 |
| **3-4** | 多种空间挑战类型未被识别或未被回应（如场景中有多个房间但战术未给出任何房间突入模式；存在开阔地但战术按走廊场景处理）。场景信息在战术中大量丢失 |
| **0-2** | 战术与场景空间类型根本性错配。例如场景主体为多层建筑，战术却在描述单层开阔地机动；或场景有明显的高低差（楼梯/阳台）但战术完全忽略垂直维度。战术相当于针对错误类型的场景生成 |

#### 8.6.3 维度二：执行效率 (Execution Efficiency)

> 评估战术行动序列是否精简、无冗余、时序最优。评估对象为行动模式本身的效率（步骤编排、并行度、路径选择），而非特定资源数量的利用率。**通用化战术不绑定机器人数量，不应以"角色负载均衡"为标准**——同一功能编组在执行阶段可由一个或多个机器人承担，编组间切换也是执行阶段的调度决策。

**评分 Checklist（逐项核查）**：

| 编号 | 核查项 | 核查方法 |
|------|--------|---------|
| E1 | **无冗余步骤** | 行动序列中是否存在可合并的相邻步骤？（如两次对同一方向的侦察可合并为一次；对同一目标的连续两次短点射可合并为一次长点射）相邻步骤去重检查 |
| E2 | **无无效等待** | 是否存在一个角色长时间等待而另一角色在执行与前者无依赖关系的步骤？如果有，该等待是否有明确的战术理由（如等待同步信号、等待侦察结果才能决策下一步）？时间线分析 |
| E3 | **并行度合理** | 独立可并行执行的行动是否被错误地串行化？（如两个方向的警戒可同时部署却写成了先后执行）不应并行的行动（如需要前一步侦察结果作为突入条件）是否被错误地并行化？依赖关系分析 |
| E4 | **路径无冗余绕行** | 机动路径是否避免了不必要的绕行？是否避免了穿越已肃清区域的冗余往返？路径效率分析 |
| E5 | **火力经济性** | 火力强度（单发精准射击/短点射/长点射/持续压制）是否与目标期望效果匹配？是否避免了"对单个疑似目标使用持续压制"的过度火力或"对集群威胁使用单发射击"的火力不足？火力-目标匹配检查 |
| E6 | **阶段衔接流畅** | Phase 之间的过渡是否有明确的完成条件和触发信号？是否避免了不必要的 Phase 间停滞？前一阶段的输出（如"走廊已肃清"）是否直接作为下一阶段的输入条件？阶段转换检查 |

**评分锚点**：

| 分数 | 判定标准 |
|------|---------|
| **9-10** | E1-E6 全部通过。并行度安排最优，无冗余步骤。火力与目标精确匹配。阶段过渡条件明确且可检测。可视为该战术模式下的步骤数下界 |
| **7-8** | E1-E5 通过。并行度基本合理，可能存在 1-2 处可优化点（如本可并行的两步骤被串行），但不影响整体时间效率。无明显的步骤冗余或火力浪费 |
| **5-6** | 存在 1-2 处明显冗余（如对同一区域重复侦察且中间无情况变化），或存在不合理串行等待。步骤数超出合理范围约 20%。火力强度存在 1-2 处不匹配 |
| **3-4** | 存在多处冗余步骤和无效等待。行动序列明显过长，并行机会被系统性忽略。火力使用存在明显浪费（对低价值目标分配持续压制火力） |
| **0-2** | 行动序列逻辑混乱，步骤缺乏合理的先后顺序。存在大量自相矛盾的时序安排。即使完成所有步骤也无法达成战术目标 |

---

#### 8.6.4 维度三：可理解性 (Comprehension)

> 评估战术描述（特别是 Intent 字段）是否精确、清晰、无歧义，使得 VLA（Vision-Language-Action）模型或人类操作员可以准确理解并执行。本维度聚焦于**语言表达的精确性**，而非战术内容的质量。

**评分 Checklist（逐项核查）**：

| 编号 | 核查项 | 核查方法 |
|------|--------|---------|
| C1 | **动作动词精确** | 每个步骤的核心动作是否使用了精确的战术动作动词？（"抵近"、"跃进"、"压制"、"侦察"、"肃清"、"占领"、"破门"、"交替掩护"）而非模糊动词（"处理"、"应对"、"搞定"、"弄一下"）动词精确性检查 |
| C2 | **目标对象明确** | 每个动作针对的目标对象是否明确？（"转角后方区域"、"房门入口方向"、"走廊北段"）而非模糊指向（"前方"、"那边"、"往里"） 对象明确性检查 |
| C3 | **执行条件清晰** | 有条件触发（如"确认安全后"、"听到突破声后"）的动作，其触发条件是否明确且可被机器人传感器检测？触发条件可行性检查 |
| C4 | **编组职责无歧义** | 每个步骤的执行主体（功能编组）是否明确且唯一？如果某步骤涉及多角色协同，各自的职责划分是否清楚、不重叠、不遗漏？角色归属检查 |
| C5 | **动作粒度一致** | 同一战术内各步骤的动作粒度是否保持一致？不应出现"步骤1：抵近转角 → 步骤2：占领整个楼层"这种粒度跳跃。粒度一致性检查 |
| C6 | **成功标准可判断** | 每个关键步骤是否隐式或显式给出了成功完成的可判断标准？（如"确认走廊无威胁"——是否说明了确认方式？视觉确认？火力侦察？传感器扫描？）完成标准检查 |
| C7 | **异常处理提示** | 关键高风险动作（突入、跨越开阔地）是否有简短的异常处理提示？（如"若突入受阻，则退至最近掩体建立火力封锁"）异常分支检查 |

**评分锚点**：

| 分数 | 判定标准 |
|------|---------|
| **9-10** | C1-C7 全部通过。每个步骤的意图可在 3 秒内被人类和 VLA 模型准确理解。所有触发条件可被传感器直接验证。关键高风险动作均包含异常分支。协同时各编组职责一目了然 |
| **7-8** | C1-C6 通过。动作动词精确，目标对象明确，角色归属清晰，粒度一致。可能存在 1-2 处"成功标准不够具体"的情况（如"确认安全"但未说明确认方式），但不影响核心执行逻辑 |
| **5-6** | 主体动作可理解，但存在 2-3 处模糊表述（如"适当距离"、"必要时"），或存在步骤间粒度不一致（1-2 处跳变）。角色切换处有暂时歧义但可推断 |
| **3-4** | 多个步骤的意图需要读者自行推断才能理解。触发条件模糊或不可被传感器检测（如"当感觉敌人放松警惕时"）。编组职责在协同步骤中混淆。粒度从原子动作跳变到宏观阶段 |
| **0-2** | 大量步骤无法确定其战术意图。动词严重模糊（大量使用"处理"、"应对"），目标对象缺失。即使人类军事专家也无法准确理解执行方式。协同时谁做什么完全不清楚 |

#### 8.6.5 维度四：粒度合规性 (Granularity Compliance)

> 评估战术 JSON 文档是否符合战术模板 v1.0 的字段规范要求。本维度聚焦于**结构性合规**（字段完整性、格式正确性、Schema 一致性、通用性规则），而非战术内容质量。

**评分 Checklist（逐项核查）**：

| 编号 | 核查项 | 核查方法 |
|------|--------|---------|
| G1 | **必填字段完整** | Tactic_ID、Tactic_Name、Mission_Phase、Tactic_Type、Objective、Description、Action_Sequence、Credibility 等模板定义的所有必填字段是否全部存在且非空？Schema 校验 |
| G2 | **字段类型正确** | 各字段的值类型是否符合模板定义？（如 Action_Sequence 必须是 Array，Credibility 必须是 Number，Semantic_Tags 必须是 Array）类型检查 |
| G3 | **Action_Sequence 结构合规** | 每个 Action 元素是否包含模板定义的全部子字段？（文字描述版：step_number + Intent + Visual_Aids；结构化描述版：step_number + Intent + Instructions）子字段完整性检查 |
| G4 | **编组化描述合规** | 执行主体是否全部使用功能性角色名（"突击组"、"压制组"、"侦察组"、"侧翼警戒组"等）而非具体机器人编号？G-T9 规则检查 |
| G5 | **通用性全规则合规** | 文字描述版是否通过 G-T1~G-T12 全部审查？结构化描述版是否通过 G-S1~G-S6 全部审查？正则 + 关键词匹配 |
| G6 | **ID 格式规范** | Tactic_ID 是否符合 `{Category}_{Number}` 格式（如 `CQB_04`）？在同一批次内是否无重复？ID 格式 + 去重检查 |
| G7 | **层级引用完整** | 如果战术设置了 Parent_Tactic 或 Sub_Tactics，引用的 Tactic_ID 是否存在于知识库中？是否避免了循环引用（A 引用 B 为子战术、B 又引用 A 为子战术）？引用完整性检查 |
| G8 | **Visual_Aids 可渲染** | Visual_Aid_Overall 和每个步骤的 Visual_Aids 是否是可被图像生成模型理解的有效描述文本？（而非"见图1"、"同步骤2"之类的占位/跳跃引用）语义有效性检查 |

**评分锚点**：

| 分数 | 判定标准 |
|------|---------|
| **9-10** | G1-G8 全部通过。所有字段完整、类型正确、格式规范，可直接通过自动化 Schema 校验。通用性审查零违规（硬约束和软约束均无违规）。Visual_Aids 描述文本完备且可直接输入图像生成模型 |
| **7-8** | G1-G7 通过。通用性审查零硬约束违规（G-T1~G-T7, G-T10~G-T11, G-S1~G-S6），可能存在 1-2 处软约束（G-T8）的可讨论项。Visual_Aids 基本可用 |
| **5-6** | G1-G4 通过（核心字段完整，编组化描述合规）。存在 1-2 处硬约束违规（如出现了 1 个具体数值或具体时间），但整体结构正确。Visual_Aids 存在 1-2 处模糊或简略 |
| **3-4** | 缺少 1-2 个必填字段，或 Action_Sequence 结构不完整（缺少 Intent 或 Visual_Aids 子字段）。存在 3-5 处硬约束违规。ID 格式不规范或有重复。Visual_Aids 多处为占位文本 |
| **0-2** | 大量必填字段缺失，JSON 结构严重偏离模板定义。通用性审查 5+ 处硬约束违规。执行主体系统性使用机器人编号而非功能编组。文件无法通过 Schema 校验 |

#### 8.6.6 维度五：图文一致性 (Text-Visual Consistency)

> 评估 Visual_Aid_Overall 和每个步骤的 Visual_Aid 描述文本是否与 Action_Sequence 中的文字描述完全对应。确保基于 Visual_Aid 生成的图像/示意图能准确传达文字战术的意图，而非产生误导。

**评分 Checklist（逐项核查）**：

| 编号 | 核查项 | 核查方法 |
|------|--------|---------|
| V1 | **空间要素对应** | Visual_Aid 描述中出现的空间要素（掩体、门、窗、走廊转角、楼梯）是否与对应 Action 的 Intent 描述中的空间要素一致？是否无多出/遗漏？空间要素交叉匹配 |
| V2 | **角色位置对应** | Visual_Aid 描述中各角色的位置、朝向、姿态是否与 Intent 描述中的角色动作一致？（如 Intent 写"压制组在掩体后方建立火力点"，Visual_Aid 中压制组是否确实在掩体后方、朝向敌方方向？）角色状态交叉验证 |
| V3 | **动作时序对应** | Visual_Aid 描述的瞬间是否与 Action 的核心动作时点匹配？（如 Action 描述"突入瞬间"，Visual_Aid 是否描绘突入动作正在发生的那一帧，而非突入后的静态站位？）时序一致性检查 |
| V4 | **角色数量一致** | Visual_Aid 中出现的角色数量是否与 Intent 中涉及的角色数量一致？没有多画或少画角色。计数检查 |
| V5 | **方向关系正确** | Visual_Aid 中的方向描述（"左侧"、"前方"、"上方"、"东北角"）是否与 Intent 中的方向描述一致？是否与 desc.json 中的空间拓扑兼容？方向一致性检查 |
| V6 | **整体图与步骤图一致** | Visual_Aid_Overall（整体战术示意图）的空间布局是否与各步骤 Visual_Aid 的空间布局一致？（不存在"同一掩体在步骤1图中位于左侧、步骤3图中位于右侧"的空间漂移）跨步骤空间一致性 |
| V7 | **视觉描述充分性** | Visual_Aid 描述文本是否包含了足够且必要的视觉细节（视角、景别、关键视觉元素的空间位置）使得生成的图像能清晰传达战术意图？而非仅有"场景示意图"等笼统描述。视觉描述充分性 |

**评分锚点**：

| 分数 | 判定标准 |
|------|---------|
| **9-10** | V1-V7 全部通过。每步 Visual_Aid 与对应 Intent 的空间/角色/时序完全对应。跨步骤空间关系恒定一致。整体图准确概括了全流程的空间逻辑和角色运动轨迹。视觉描述丰富度足以生成专业级战术示意图 |
| **7-8** | V1-V6 通过。角色位置、空间要素、时序、方向均一致。V7 基本满足（图像可清晰传达战术意图），可能存在 1-2 处视觉细节不够丰富（如未指定景别），但不影响核心信息的视觉传达 |
| **5-6** | 核心一致性可接受，但存在 1-2 处不一致。例如步骤中 Intent 提到 3 个角色但 Visual_Aid 只描述了 2 个；或方向描述有轻微偏差。不影响核心战术意图的视觉传达，但细节不可靠 |
| **3-4** | 多处不一致。角色位置与 Intent 描述在多步骤中矛盾，空间布局在不同步骤间系统性漂移。基于 Visual_Aid 生成的图像会误导执行者对空间关系的理解 |
| **0-2** | Visual_Aid 与 Intent 严重脱节。Visual_Aid 描述的是错误的场景要素、错误的角色数量、错误的空间关系。或者 Visual_Aid 为空或为占位文本（"如图"、"见上文"） |

#### 8.6.7 维度六：军事可行性 (Military Feasibility)

> 评估战术方案是否遵循已确立的军事原则和战术条令，在真实的物理条件和敌方对抗条件下是否可执行。**注意：通用化战术不规定兵力数量和具体编成**——兵力配置是执行阶段根据可用机器人数量进行的场景绑定决策。本维度评估的是行动模式（pattern）层面的军事合理性，而非特定兵力部署方案的正确性。

**评分 Checklist（逐项核查）**：

**子维度 6.1：编组互助关系 (Mutual Support Relations)**

*通用化战术使用功能性角色（突击组/压制组/侦察组/侧翼警戒组等），评价关注的是这些角色之间是否形成了相互支援的行动模式，而非固定的人员配置比例。*

| 编号 | 核查项 | 核查方法 |
|------|--------|---------|
| M1 | **功能编组互助闭环** | 战术中的各功能编组之间是否形成了相互支援的关系？是否存在某个角色被其他角色掩护/侦察/压制的对应关系？每个承担风险暴露任务的角色（如突入的突击组）是否至少有一个其他角色为其提供掩护或侦察支持？编组互助关系检查，不检查兵力数量配比 |
| M2 | **无孤立行动角色** | 任何功能编组是否被要求在无掩护/无侦察支持/无退路保障的情况下独自深入未肃清区域？是否存在"一个角色承担全部风险、其他角色仅旁观"的不平衡模式？孤立行动检查，不要求显式保留"预备队" |
| M3 | **功能编组数量自洽** | 战术要求的行动模式在逻辑上最少需要几个独立执行体？如果战术涉及同时执行的两个不同位置的动作（如同步突入两个门），则隐式需要至少两个独立执行体——战术是否在逻辑上自洽？执行阶段是否能为此分配足够的机器人？角色数量自洽检查 |

**子维度 6.2：空间控制 (Spatial Control)**

| 编号 | 核查项 | 核查方法 |
|------|--------|---------|
| M4 | **关键地形占据** | 战术是否识别并优先控制了场景中的关键地形类型（制高点、扼守点、转角控制点、楼梯口）？关键地形的控制逻辑是否在推进之前完成？关键地形检查 |
| M5 | **火力扇区无冲突** | 多角色建立的火力扇区是否存在交叉火力中的友军火力风险（Fratricide Risk）？角色的机动路径是否穿越了友军已建立的火力扇区？火力扇区交叉检查 |
| M6 | **机动路径安全** | 机动路径是否避开了敌方最可能的火力杀伤区（基于 inferred_threats 和 cover_assessment 推断）？如果必须穿越杀伤区，是否安排了压制掩护或遮蔽手段？路径-杀伤区分析 |
| M7 | **死角清理** | 是否对空间中可识别的一切视线死角类型（转角的对侧、掩体后方、门后空间、垂直方向/上方夹层）安排了清理或确认手段？死角覆盖检查 |

**子维度 6.3：时间与节奏 (Tempo & Timing)**

| 编号 | 核查项 | 核查方法 |
|------|--------|---------|
| M8 | **节奏与威胁匹配** | 高威胁区域是否采用了更谨慎的推进节奏模式（切片侦察、交替掩护）？低威胁区域是否避免了过度拖慢节奏以致丧失突然性？节奏-威胁匹配 |
| M9 | **同步可行性** | 标注为"同步"的动作是否在物理上可同时执行？（不共享同一条狭窄通道、不在同一掩体后方发生空间冲突）同步冲突检查 |
| M10 | **OODA 循环完整** | 战术的推进节奏是否给执行体留出了足够的观察-判断-决策-行动循环时间？是否出现了不合理的期望（如要求刚完成机动的角色立即做出精确射击判断）？OODA 可行性检查 |

**子维度 6.4：对抗合理性 (Adversarial Realism)**

| 编号 | 核查项 | 核查方法 |
|------|--------|---------|
| M11 | **敌方响应考虑** | 战术是否考虑了敌方的可能响应模式——固守待援、主动反击、转移人质、设伏？是否针对敌方最可能行动方案（EMLCOA）和最危险行动方案（EMDCOA）留有应对弹性（而非硬编码的预案）？EMLCOA/EMDCOA 覆盖 |
| M12 | **不依赖敌方错误** | 战术的成功是否依赖于不合理的假设——"敌方不会从背后出现"、"敌方不会射击"、"敌人一定在某个位置"？所有关键假设是否被显式标注并给出了假设不成立时的应对方向？假设合理性检查 |
| M13 | **信息不确定性处理** | 对场景中标记为 uncertain 的信息（如"人质位置不确定"），战术是否采用了稳健策略（先侦察确认再行动）？而非假设最优情况（假设人质就在某个房间直接突入）径直行事？不确定性处理 |

**子维度 6.5：战术原则遵循 (Principles of War Compliance)**

以下核查项对照北约/美军九大战争原则中与行动模式层面直接相关的五项进行评估。**注意：Mass（集中）和 Economy of Force（节约兵力）原则在通用化战术中评估的是"是否在决定性地点集中了优势行动"，而非是否部署了固定数量的兵力。**

| 编号 | 核查项 | 对应原则 | 核查方法 |
|------|--------|---------|---------|
| M14 | **突然性 (Surprise)** | Surprise | 战术是否利用了突然性？（时机选择、突入方向选择、声东击西、利用环境条件掩护行动） |
| M15 | **简单性 (Simplicity)** | Simplicity | 战术模式是否足够简单以至于在高压和通信受限条件下仍可执行？是否避免了不必要的复杂协同分支和不必要的多条件嵌套？ |
| M16 | **集中 (Mass)** | Mass + Economy of Force | 战术是否在决定性地点（Decisive Point——如突入点、关键转角）集中了优势行动力量？是否避免了在全空间均匀摊薄注意力和行动力度？**注意：评估的是"行动重心"而非"兵力数量"——一个角色在正确时机对关键点施加强大影响，即体现了集中原则** |
| M17 | **安全 (Security)** | Security | 是否保护了自身的脆弱面——后方路径、退路、通信不被敌方切断或伏击？战术是否考虑了"如果此步骤失败，如何安全撤出？" |
| M18 | **统一指挥 (Unity of Command)** | Unity of Command | 多人协同步骤是否有明确的协调角色？同步点的主导方是否明确？如果协同失败，谁负责决策和重新同步？

**评分锚点**：

| 分数 | 判定标准 |
|------|---------|
| **9-10** | M1-M18 全部通过。五个子维度均无缺陷。战术展示了主动利用敌方脆弱点的意识，考虑了敌方最可能和最危险两种响应模式。关键假设均被显式标注。编组间互助关系稳固，每个承担风险的角色背后都有支援角色。可视为该场景类型下的条令最佳实践，具备作为正面范例（Few-Shot Example）的质量 |
| **7-8** | M1-M13 全部通过（互助关系/空间/时间/对抗四个子维度无缺陷）。战术原则 M14-M18 中可能有 1-2 条在该场景类型下无法充分体现（如狭小空间内突然性本身就受限，或极简场景下简单性自然满足）。核心战术逻辑坚实，可直接执行 |
| **5-6** | 存在 1-2 处军事缺陷但可修正。例如：某角色在特定步骤中短暂缺乏直接支援但在低威胁区可接受；或某条机动路径部分穿越了潜在杀伤区但有压制掩护措施。需人工指挥官审核并确认风险可接受后执行 |
| **3-4** | 存在多处军事原则违反。例如：角色被要求在没有支援的情况下深入未肃清区域、火力扇区存在未解决的潜在冲突、敌方响应被系统性忽略。战术在真实对抗中失败概率高。需 M3 定向修正 |
| **0-2** | 战术违反基本军事常识。例如：要求角色背对未肃清区域机动、交叉火力配置导致必然的友军火力、要求"在不被发现的情况下从开阔地接近敌方观察哨"。执行必然导致任务失败或不可接受的损失 |

#### 8.6.8 综合评分与质量分级

综合 6 个维度的评分：

$$Q(T) = \frac{1}{6} \sum_{i=1}^{6} q_i(T)$$

**否决条件（Veto Criteria）**：以下任一条件触发，直接判定为 L 级（不入库），无论综合加权评分如何：

| 代码 | 否决条件 | 说明 |
|------|---------|------|
| **V1** | 维度六（军事可行性）评分 < 3.0 | 违反基本军事常识的战术即使其他维度满分也没有入库价值 |
| **V2** | 维度一（场景适配度）评分 < 3.0 | 战术与场景严重脱节，相当于针对错误场景生成 |
| **V3** | 维度四（粒度合规性）评分 < 3.0 | 5+ 处硬约束违规或大量字段缺失，格式不可修复 |

**质量分级标准**：

| 质量等级 | 综合评分 | 附加条件 |
|----------|---------|---------|
| **H**（高质量） | $Q \geq 8.0$ | 维度六（军事可行性）≥ 7.0，且无任何单一维度 < 4.0 |
| **M**（中质量） | $6.0 \leq Q < 8.0$ | 无否决条件触发，无任何单一维度 < 4.0 |
| **L**（低质量） | $Q < 6.0$ 或任意否决条件触发 或任意单一维度 < 4.0 | 不入库，仅保留用于训练数据筛选和错误模式分析 |

> **H 级附加条件说明**：军事可行性维度单独设 ≥ 7.0 的门槛是因为——无论其他维度评分多高，军事可行性存在明显缺陷的战术不配标记为"高质量可直接使用"。6 个维度的平均分会掩盖某一维度的严重短板。

#### 8.6.9 LLM 评价 Agent 使用指南

1. 逐维度读取对应的 Checklist（§8.6.2 - §8.6.7），对每个核查项进行"通过/不通过/部分通过"判定

3. 对每个不通过或部分通过的核查项，记录违规的具体位置（Action_Sequence 索引、字段路径）和扣分理由
4. 检查否决条件 V1-V3
5. 计算综合评分并判定质量等级


## 九、战术知识库组织

### 9.1 知识库层次结构

```
战术战法知识库
├── Layer 1: 场景库 (Scene Library)
│   ├── 大场景源文件: 原始 USDA 场景文件（data/raw/）
│   ├── 场景元数据: Phase 0 解析输出（data/processed/phase0/）
│   ├── 全局场景理解: Phase 1 LLM 解析输出（data/processed/phase1/）
│   ├── 子场景定义: Phase 2 子场景划分 JSON（data/processed/phase2/）
│   └── 子场景文件: Phase 3 简化 scene_cubes.json + desc.json（data/processed/sub_scenes/SS_XX/）
│
├── Layer 2: 战术库 (Tactics Library)
│   ├── 文字描述版: 粗粒度，自然语言描述 + 视觉辅助（tactics/text_version/{H,M,L}/）
│   ├── 结构化描述版: 细粒度，结构化指令 + 参数占位符（tactics/struct_version/{H,M,L}/）
│   ├── 质量标记: H/M/L 三级
│   └── ChromaDB 索引: 三个 Collection —— `tactics_text`、`tactics_struct`、`pdf_chapters`
│
└── Layer 3: 元数据 (Metadata)
    ├── 质量评分: 各版本的 6 维度评分（嵌入在 JSON `_metadata` 字段中）
    ├── 生成记录: 生成模式、迭代轮次、参考来源、子场景来源
    ├── 关联关系: 战术间父子关系、场景-战术映射、子场景-子场景衔接
    └── 审计信息: 创建时间、审核状态
```

> **ChromaDB 集成说明**：知识库的后端存储为 ChromaDB PersistentClient（持久化目录 `data/processed/chroma_db/`）。每条战术通过 `EmbeddingClient` 生成嵌入向量后写入对应的 Collection（`ingestion.ingest_tactic()`），元数据（tactic_id/tactic_name/mission_phase/tactic_type/quality_level/sub_scene_id/generation_mode）作为 ChromaDB metadata 存储。H/M/L 分级通过 `quality_filter` 参数控制入库范围（如仅入库 H 和 M 级）。

## 十一、实现顺序与验证方案

### 11.1 实现顺序

1. **Phase 0 解析器**（已完成）：流式 USDA 解析 + 世界坐标计算 + 楼板面证据楼层推断（classify_fragments→extract_floor_planes→_merge_adjacent_candidates→_build_stories_from_slabs, N-1逻辑: 4 slabs→3 stories）+ 几何特征提取（geometry_features: 墙面线/房间/走廊/密度楼梯）+ 几何模式检测（纯几何输出，不做语义分类）。8 步流水线：stream_parser → prim_builder → transform_accumulator → z_layer_analyzer → pattern_detector → geometry_features → floor_evidence → material_groups。楼层检测参数通过 `FLOOR_DETECTION_PARAMS` 可配置
2. **Phase 1 LLM prompt**（已完成）：数据精简（12 个代表性 Cube + 空间密度分布 + 材质统计），LLM 生成全局场景理解。输出约束：≥8 spatial_layouts + ≥8 tactical_features + 300-500 字分段描述。支持 `--task` 参数引导场景理解方向
3. **Phase 2 LLM prompt + 去重**（已完成）：基于全局理解划分子场景边界，含 reviewer 审核模块。支持 `--task` 参数，LLM 将总任务分解为每个子场景的 task_hint。LLM 驱动子场景战术等价去重（§6.5），在 Phase 2 runner（主入口）和 Phase 3 pipeline（安全网）两处调用
4. **Phase 3 裁切+简化+语义标注**（已完成）：3a cropper → 3b simplifier（分类初筛→同类型合并→模式折叠→归一化）→ 3c LLM 语义标注 → validator 自动校验
5. **Phase 4 GTKG-CM**（已完成）：M1→M2→Stage 0 穷举枚举→M3（逐概念精炼）→M4 完整链路。M1 一致性验证，M2 双层自适应策略（BGE 嵌入+余弦相似度+LLM 自评），Stage 0 参考分批注入+字符二元组去重+迭代穷举（最多 15 个战术概念），M3 A_gen(含 seed_concept 扩展)↔A_review 辩论式迭代（含 Few-Shot 示例注入），M4 六维度评价 + H/M/L 分级
6. **知识库集成**（已完成）：ChromaDB 向量存储 + 嵌入客户端 + 战术入库/检索接口
7. **批量处理**（已完成）：`PipelineRunner.run_all()` 全流水线编排，`run_phase4_batch()` 批量子场景处理

## 十二、总结与展望

### 12.1 概念模型总结

本文提出的生成式战法战术知识生成概念模型（GTKG-CM）包含以下核心贡献：

1. **完整工程流水线**：从 928MB Blender 导出 USDA 大场景出发，通过 Phase 0（流式解析）→ Phase 1（全局理解）→ Phase 2（子场景划分）→ Phase 3（几何简化）→ Phase 4（GTKG-CM 战术生成）五阶段流水线，实现"大场景到原子级战术"的端到端处理。

2. **场景驱动的战术知识生成范式**：通过 USD 标准三维场景文件作为统一输入格式，将场景的全部信息嵌入单个 USDA 文本文件。提出编组化描述原则、场景无关性原则（三重不变性——旋转不变性+物体功能不变性+编队规模不变性）、相对关系描述原则和双版本输出机制，确保战术的通用性与可执行性。

3. **自适应生成策略**：提出"检索置信度 + LLM 自评"双层模式切换机制，在检索增强生成与自主生成之间实现动态选择，兼顾已有知识利用与新场景应对能力。

4. **多智能体协作的迭代精炼框架**：设计了基于辩论式迭代精炼的多智能体协作协议，通过生成 Agent 与审查 Agent 的结构化反馈循环，在保证生成质量的同时控制迭代效率（最大3轮，收敛阈值 0.5 分）。

5. **结构化的质量评价体系**：定义了6维度评价 rubric 和质量分级标准（H/M/L），支持金标准数据集校准，为战术知识的质量管理提供了可操作的评估框架。

## 附录B：USD/USDA 格式与战术场景表示

### B.1 USD 格式概述

**USD（Universal Scene Description）** 是 Pixar Animation Studios 开发的开放源代码三维场景描述框架，现由 OpenUSD 联盟（Pixar、NVIDIA、Apple、Adobe、Autodesk）共同维护。USD 在机器人仿真领域得到 NVIDIA Isaac Sim 等平台的广泛支持。

USD 支持多种文件格式，GTKG-CM 采用 **USDA（ASCII 文本格式）**：

| 格式 | 全称 | 编码 | LLM 可读性 | 适用场景 |
|------|------|------|-----------|----------|
| `.usda` | Universal Scene Description ASCII | 纯文本 | 高 | LLM 直接读取、人工编辑 |
| `.usdc` | Universal Scene Description Crate | 二进制 | 低 | 高效存储与传输 |
| `.usdz` | USD Zipped | 压缩包 | 低 | 打包分发（含纹理等依赖） |

### B.2 USDA 核心概念

USDA 文件以层级**场景图（Scene Graph）** 组织内容，核心概念如下：

| 概念 | 说明 | 示例 |
|------|------|------|
| **Prim** | 场景图中的基本节点，可以是几何体、变换、材质、光源等 | `def Xform "Room_01"` |
| **Property** | Prim 的属性，包括数值、字符串、数组等 | `float3 xformOp:translate = (0, 0, 0)` |
| **Xform** | 变换节点，定义位置、旋转、缩放 | `double3 xformOp:translate = (5.0, 2.0, 0)` |
| **Mesh** | 网格几何体，通过顶点和面定义三维形状 | `int[] faceVertexCounts = [4, 4, ...]` |
| **Scope** | 组织节点，不改变变换层级，用于逻辑分组 | `def Scope "Doors"` |
| **Material** | 材质定义，绑定到 Mesh | `def Material "Wall_Paint"` |
| **Light** | 光源（DistantLight/SphereLight/DomeLight 等） | `def DistantLight "Sun"` |
| **customData** | 自定义元数据字典，可嵌入任意语义信息 | `customData = {string mission = "recon"}` |

### B.5 USDA 到战术空间的映射

LLM 读取 USDA 文件后，建立从 USD 要素到战术空间的映射。以下标注每条映射的当前可用性（基于 Blender 导出无语义标注的现实）：

| USD 信息 | 提取方式 | 战术空间含义 | 当前可用性 |
|----------|----------|-------------|-----------|
| `Building.floors` custom 属性 | 直接读取 | 建筑层数 → 清剿层级规划 | **不可用** — Blender 无此标注，由 Phase 0 从楼板面证据+N-1逻辑推断，Phase 1 确认 |
| `Openings` Scope 中的 Xform | 遍历 Prim 树 + `opening_type` 标注 | 所有可用入口 → 突入点候选集 | **不可用** — Blender 无此标注，由 Phase 3c LLM 从 door/window Cube 识别 |
| 各 Xform 的 Z 坐标 | 坐标值分析 | 垂直空间分布 → 楼层推断 | **可用** — Phase 0 楼板面证据（floor_evidence） |
| `Cover_*` Xform 的 customData | `cover_type` 属性读取 | 掩体类型与位置 → 掩护规划 | **不可用** — 由 Phase 3b 启发式 + Phase 3c LLM 识别和评估 |
| 各 Prim 的空间遮挡关系 | 坐标距离计算 | 视线/盲区 → 火力覆盖规划 | **间接可用** — Phase 3c LLM 从几何布局推理（`inferred_threats`） |
| `DistantLight.direction/intensity` | 属性读取 | 光照条件 → 昼夜与阴影方向 | **可用** — Phase 0 解析 `DomeLight` / `DistantLight` |
| `entity_type=enemy` 标记 | customData 读取 | 敌方位置、威胁等级、武装 → 战术规避或优先清除 | **不可用** — Blender 无此标注；Phase 3c LLM 仅推理潜在威胁位置（`inferred_threats`），非已知敌方标记 |
| `entity_type=hostage` 标记 | customData 读取 | 人质位置、状态、人数 → 搜索优先级与行动约束 | **不可用** — Blender 无此标注；人质信息目前完全缺失，需指挥官在任务描述中提供 |
| 简化USDA `tactical_type` | customData 读取 | Phase 3 输出的语义标签 → 直接映射到战术空间 | **可用** — Phase 3b Python 写入，Phase 3c LLM 纠错 |

### B.6 Blender 导出 USDA 与理想语义 USDA 的关键差异

| 特征 | 理想语义标注 USDA | Blender 导出实际 USDA |
|------|-------------------|----------------------|
| Prim 命名 | 语义化（`Wall_North`、`Main_Entrance`） | 自动生成（`Group_24__31772_278`） |
| customData | 丰富（entity_type、cover_type、opening_type） | 无 |
| 层级组织 | 逻辑分组（Openings、Known_Entities 等 Scope） | 扁平化 Xform 嵌套 |
| 文件大小 | 小（数十KB） | 大（数百MB，含完整顶点数据） |
| LLM 可读性 | 直接可读 | 需 Phase 0-3 预处理 |
| 语义信息 | 显式标注 | 需启发式推断 + LLM 判定 |
